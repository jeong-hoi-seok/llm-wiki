# LLM Wiki와 옵시디언으로 프론트엔드 작업 자동화하기

## 한 줄 요약

멀티 프로젝트에서 공통 컨벤션을 각 프로젝트 `docs/`에 복제하면 중복·누락·동기화 비용이 생긴다. 공통 지식은 별도 Git 저장소(공통 LLM-WIKI Store) 한 곳에서 관리하고, 각 프로젝트는 `AGENTS.md` 진입점 + `$HOME` 부트스트랩 clone + 심볼릭링크로 그 Store를 참조한다. 프로젝트 고유 맥락은 각 프로젝트 위키에 남긴다.

> 발행본: [멀티 프로젝트의 관리 리소스를 줄이는 계층적 LLM-WIKI 방법론](https://www.linkedin.com/pulse/멀티-프로젝트의-관리-리소스를-줄이는-계층적-llm-wiki-방법론-회석-정-xmmlc) (정회석). 공개 저장소: https://github.com/jeong-hoi-seok/llm-wiki

## 배경: 단일 프로젝트 하네스의 한계

신입 A씨가 입사했을 때 사내 개발 컨벤션은 Notion에 정리돼 있었다. A씨는 이를 학습했지만, AI에게 커밋을 지시했을 때 AI가 커밋 컨벤션을 몰라 타이틀이 컨벤션과 어긋났고 리뷰에서 close됐다.

A씨는 프로젝트 안에 `docs/`를 만들어 컨벤션 문서를 넣고, root `AGENTS.md`에 "`docs/`에 관련 문서가 있다"고 표기해 학습을 지시했다. AI가 그 문서를 읽고 컨벤션대로 작업하면서 효과가 컸다.

문제는 신규 프로젝트였다. 새 repo를 세팅하며 `docs/`에 컨벤션 문서를 넣다가 **하나를 누락**해, 다시 컨벤션 미준수로 리뷰에서 close됐다. 여기서 한계가 드러난다.

- 하네스(프로젝트 내 `docs/`) 방식은 단일 프로젝트엔 효과적이지만, **프로젝트가 여러 개이고 공통 컨벤션이 있으면 관리 리소스와 휴먼 에러**가 생긴다.
- 공통 컨벤션이 바뀌면 **모든 프로젝트의 `docs/`를 일일이 갱신**해야 한다. 비효율적이다.

## LLM-WIKI란?

LLM-WIKI는 Andrej Karpathy가 제안한 지식 저장·아카이빙 방법이다. 공부한 내용, 회의록, 작업 로그 등 모든 문서를 정리해 넣는 지식 저장소다.

배경에는 AI의 맥락 파악 문제가 있다. AI는 새 세션마다 Stateless 특성으로 맥락을 잊는다.

### 맥락이 왜 중요한가 — useEffect 케이스

A씨가 `useEffect`로 기능을 구현하다, 내부에서 쓰는 값을 의존성 배열에 넣으면 무한 루프가 도는 상황을 만났다. 그 값을 내부에서 꼭 써야 했기에 의존성에서 빼고 `eslint-disable` 주석으로 린트 경고를 막은 뒤 마무리했다. 이 결정은 코드 리뷰에서 설명·승인돼 머지됐다.

몇 달 뒤 B씨가 AI에게 전체 코드 리뷰를 요청한다. AI는 그 구간을 보고 "의존성에 값을 넣지 않고 린트 무시 주석으로 우회했다"는 결함으로 판단한다 — **당시 결정의 배경(맥락)을 모르기 때문**이다.

A씨가 그 결정의 배경을 어딘가에 로그로 남겼고 AI가 그것을 학습할 수 있었다면, B씨의 개선 작업에서 AI는 주석을 함부로 지우지 않았을 것이다. LLM-WIKI는 그런 **기억의 로그를 기록해 AI가 맥락을 파악**하게 한다. 즉, 대량 아카이브에서 필요한 부분을 탐색해 빠르게 학습하고, 새 세션에서도 이어가게 하는 문서 아카이빙 기술이다.

## 멀티 프로젝트 환경에 적용하기

LLM-WIKI를 안다고 문제가 바로 풀리진 않는다. 컨벤션 문서를 각 프로젝트 `docs/`에 나눠 두면 저장 형식만 바뀔 뿐 중복·누락·동기화 문제는 그대로다. 핵심은 **LLM-WIKI 자체가 아니라 이를 멀티 프로젝트에 어떤 구조로 적용할 것인가**였다.

기존 공통 컨벤션은 Notion에 있었지만, 코딩 에이전트가 Notion을 직접 참조하려면 MCP·API 같은 별도 연동이 필요하다. 우리가 원한 건 **추가 연동 없이 로컬에서 읽고, 변경 이력·버전을 코드처럼 관리**하는 구조였다.

그래서 공통 문서를 특정 프로젝트에 종속시키지 않고 **별도 Git 저장소(공통 LLM-WIKI Store)** 로 분리했다. 각 프로젝트에 복제하는 대신 모든 프로젝트가 하나의 Store를 참조한다. 공통 문서는 Store에서 한 번만 관리하고, 변경 시 Store 원본만 갱신하면 각 프로젝트는 최신 내용을 동기화한 뒤 읽는다.

## 개별 프로젝트도 위키가 필요하다

공통 Store는 여러 프로젝트가 공유하는 지식 아카이브다. 하지만 공통 지식만으로는 **프로젝트 맥락**(무엇을 했고 왜 그렇게 결정했는지)을 알 수 없다 — 그건 공통 Store에 기록되지 않기 때문이다.

그래서 공통 문서는 Store에, **프로젝트 고유 작업 기록·결정 맥락은 각 프로젝트 위키**에 남긴다. 이렇게 분리하면 공통 Store가 프로젝트별 정보로 비대해지지 않고, AI도 현재 프로젝트 맥락만 일정한 방식으로 탐색한다. 공통 Store에는 **프로젝트별 위키를 어떤 구조로 만들고 언제 갱신할지에 대한 관리 가이드**도 함께 둬, 각 프로젝트 AI가 이를 참조해 위키를 자동 생성·관리한다.

### 개별 프로젝트 위키 구조

| 폴더 | 역할 |
|---|---|
| `raw` | 회의록·작업 메모 등 가공 전 원본 |
| `summaries` | raw 핵심 요약 |
| `decisions` | 주요 결정과 배경 |
| `worklog` | 작업·변경 이력 |

> 이 본진(`~/.llm-wiki`)의 운영 가이드는 [[프로젝트 맥락 위키 작성 가이드]]에 정리돼 있다.

## 프로젝트에서 Store 위키를 참조하는 법

서로 다른 저장소의 프로젝트들이 공통 위키를 참조하는 방법이 다음 문제였다. 각 프로젝트에 `AGENTS.md`를 두고 공통 위키의 저장소 주소·로컬 경로·읽는 순서를 지시한다.

AI는 작업 시작 시 `AGENTS.md`를 확인해 지정 경로에서 공통 위키를 찾고, 없으면 clone, 있으면 기존 것을 쓴다. 이후 **모든 문서를 한꺼번에 읽지 않고** 개발 원칙·공통 컨벤션처럼 필수 문서만 먼저 컨텍스트에 넣는다. 추가 정보가 필요하면 공통·프로젝트 위키에서 찾고, 위키에도 없을 때만 외부 자료를 탐색한다.

`AGENTS.md`를 진입점으로 둔 이유는 **팀원마다 쓰는 AI 도구가 다를 수 있기 때문**이다. 도구와 무관하게 같은 지시·탐색 절차를 따르게 한다. 사용자는 프로젝트마다 위키를 직접 설치하거나 문서를 매번 지정하지 않아도 된다.

> 작성 기준은 [[AGENTS.md 작성 가이드]]가 단일 기준(SSOT)이다.

## 공통 경로의 필요성

가장 큰 병목은 **경로 지정**이었다. `AGENTS.md`에 위키 경로를 적어야 하는데, 룰 없이 "사내 GitHub에서 클론하세요"라고 하면 누구는 `project/` 안에, 누구는 `User/local`에 저장한다. 그러면 선언한 경로가 무의미해진다. 기술마다 트레이드오프를 따진 끝에 **부트스트랩 최초 clone + 운영체제 무관 경로 지정**을 룰로 잡았다.

### 대안 비교

| 대안 | 판정 | 이유 |
|---|---|---|
| ① 서브모듈 | ❌ | 원본 repo가 프로젝트 안에 그대로 들어옴 — 1만 파일 위키를 프로젝트마다 복제하는 꼴 |
| ② 각자 경로를 환경변수로 관리 | ❌ | 사용자마다 클론 경로를 env에 등록·관리 → 온보딩 비용 증가 |
| ③ git URL network fetch | ❌ | 로컬 저장 불필요·URL 절대경로·항상 최신이나, 호출마다 네트워크 왕복 + 캐싱 불가 → 리소스 과다 |
| ④ git URL clone 후 공통 경로 | ✅ | 로컬에 공통 경로로 저장해 사용 |

채택안(④)의 두 축:

- **지속적 동기화** — wiki index에 버전을 명시하고, 위키를 읽을 때 최우선으로 "로컬 위키 버전" vs "git 릴리즈 버전"을 비교해 다르면 먼저 pull 후 학습하도록 지시.
- **경로 통일** — Mac·Windows 모두 쓸 수 있는 `$HOME/`으로 지정해 각 OS의 USER 폴더 경로에 들어가게 함.

## 또 다른 문제: 외부 참조

공통 위키를 `$HOME` 아래 두면서, 각 프로젝트 AI가 위키를 읽으려면 작업 디렉터리 **밖**의 경로에 접근해야 했다. 코딩 에이전트마다 외부 접근 권한·정책이 달라 차단되거나 매번 승인이 필요할 수 있다.

이를 해결하려고 **프로젝트 내부에 공통 위키를 가리키는 심볼릭링크**를 생성했다. 실제 위키는 `$HOME` 공통 경로에서 한 번만 관리하지만, 프로젝트에서는 내부 링크로 접근한다. AI 입장에선 프로젝트마다 **동일한 내부 경로**를 진입점으로 쓰고 지시문도 그 경로 기준으로 통일된다. 복제 없이 일관된 방식으로 외부 LLM-WIKI를 참조하는 구조다.

> 심볼릭링크는 절대경로라 머신마다 깨지므로 소비 프로젝트 `.gitignore`로 제외하고 부트스트랩으로 재생성한다. 실체 repo(`~/.llm-wiki`)는 정상 push한다. 상세는 [[AGENTS.md 작성 가이드]].

## 문서 작성은 Obsidian, 버전 관리는 Git

편집 도구로 Obsidian을 골랐다. Markdown 기반이라 변환 없이 Git에서 바로 관리되고, 그래프 뷰로 문서 간 연결을 보며 탐색·수정하기 편하다. 각 문서 상단 Frontmatter에 `scope` 등 메타데이터를 써, 공통/플랫폼/프로젝트 문서를 분류하고 AI가 필요한 문서만 선별해 읽게 한다.

Obsidian이 작성·탐색을 맡으면 Git은 저장·변경 이력을 맡는다. 언제 왜 수정됐는지 추적하고 필요하면 이전 버전으로 되돌린다. 즉 **작성·분류는 Obsidian, 원본·버전은 Git**이다.

## 적용 결과

`AGENTS.md`만 설정된 환경에서 "프로젝트 셋팅해줘"라고만 명령해도, AI가 지정 문서를 읽고 학습해 컨벤션대로 셋팅한다. 구조와 지시문은 공개 저장소(https://github.com/jeong-hoi-seok/llm-wiki)에 두었다.
