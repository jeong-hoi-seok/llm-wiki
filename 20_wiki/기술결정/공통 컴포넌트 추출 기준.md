# 공통 컴포넌트 추출 기준

> When to Extract a Common Component — 추출·유지·폐기 판단 기준. 상위: [[index]]

## TL;DR
공통 컴포넌트는 **영구 자산이 아니다.** 만들기·유지·폐기를 세 축으로 판단한다:

| 판단 축 | 질문 |
|---|---|
| **재사용 횟수** | 실제로 여러 곳에서 쓰이나? (이론적 가능성 ❌) |
| **의존 안정성** | 기대고 있는 하위 컴포넌트가 안 흔들리나? |
| **유지비용** | 공통화 이점 > 앞으로 감당할 변경·검증 비용? |

→ 정답을 정하는 게 아니라 **판단 기준을 팀이 공유**하는 게 핵심.

## 생애주기 한눈에
```
탄생 ──▶ 사춘기 ──▶ (독립 미정) ──▶ 결합 ──▶ 죽음
만들까?   비대화     경계선 유틸      템플릿    deprecated
```
각 단계마다 던질 질문이 다르다. 아래 단계별 정리.

---

## 1️⃣ 탄생 — 만들기 전에
**만들기 전 3가지 질문:**
- [ ] 이미 있는 컴포넌트 **조합**으로 해결 못 하나?
- [ ] 지금은 "공통 아니다"라고 둬도 괜찮은 시점 아닌가?
- [ ] 나중에 공통으로 옮겨도 문제없는 구조인가?

**🚫 섣부른 공통화 신호 (보류해라):**
| 신호 | 이유 |
|---|---|
| 사용처가 적다 (예: 2곳) | 재사용 이점이 아직 작음 |
| **deprecated 예정 컴포넌트에 의존** | 하위 변경 시 영향 범위 예측 불가 |
| 이점 < 향후 유지비용 | 만드는 순간 부채가 됨 |

> 💡 핵심 전환: "공통 만들 수 **있나**?"가 아니라 "언제 **잘 만들었다**고 할 수 있나?"

## 2️⃣ 사춘기 — 비대화 경계
한 컴포넌트가 역할을 계속 흡수하면 책임이 흐려지고, **옵션 추가마다 본체를 수정**하게 된다.

- 예) 보험 상품 선택 카드(`CoverageSelectCard`)가 흡수한 것:
  바텀시트 열기/닫기 · iOS body height 대응 · 드래그 인터랙션 · 애니메이션 제어
- 증상: 새 요구사항마다 컴포넌트 본체를 또 건드림 → [[응집도|응집도]]↓·[[결합도|결합도]]↑

> 질문은 "이거 **나쁜가**?"가 아니라 **"역할을 언제·어떤 기준으로 나눌까?"**
> → 헤드리스/조합으로 분리 검토 ([[헤드리스 컴포넌트]])

## 3️⃣ 독립 미정 — 경계선 유틸
작고 도메인 무관한 유틸(예: `DelayRender` — 일정 시간 후 children 렌더)은 **공통이라 하기도, 아니라 하기도 애매**한 경계선.

팀이 합의할 기준:
- 몇 번 반복부터 "공통"으로 볼 것인가?
- 실제 사용 빈도 vs 관리 비용을 어떻게 저울질?
- **"지금은 공통이 아니다"라고 말할 시점**은 언제인가?

> "언제 만드나"보다 **"언제까지 공통으로 유지하나"**를 묻게 만드는 사례.

## 4️⃣ 결합 — 템플릿
`단일 컴포넌트(자유)` → `느슨한 조합(관계 시작)` → `템플릿(자유 아님)`.
**템플릿 = 재사용 도구가 아니라 "변경하지 않기로 합의한 결정 집합".**

| ✅ 그린 라이트 (결합해도 됨) | 🚫 레드 라이트 (아직 아님) |
|---|---|
| 같은 조합이 복붙 반복됨 | 화면마다 예외가 계속 생김 |
| 조합의 주체가 명확 | 검증·케이스 부족, 불확실한 미래에 베팅 |

## 5️⃣ 죽음 — deprecated
**deprecated는 실패 선언이 아니라 비용 절감 선택.** 대체 컴포넌트를 함께 제시한다.

죽음의 신호:
- 아무도 안 찾음 (잊혀짐)
- 외부 정책 요인 (예: 결제 수단 변경)
- 로직이 너무 복잡해 손대기 두려움

> deprecated의 시작 = "사용 안 함"이 아니라 **"의도적으로 선택되지 않음"**. 팀 기본 선택지에서 빠지는 순간.

---

## 해결 방향 — 가이드 ↔ 자유 양립
일관성(가이드)과 커스터마이징(자유)을 동시에 → [[헤드리스 컴포넌트]]
```
Headless (로직 + 접근성만 공통)
 ├─ 기본 UI 버전  (디자인 가이드 준수)
 └─ 자유 UI 버전  (서비스 특성 커스텀)
```
안정성·접근성은 공통, 디자인은 유연 허용.

## 실무 적용 — PR·리뷰 체크리스트
- [ ] 새 공통 컴포넌트? → 탄생 3질문 통과했나
- [ ] 기존 공통 사용? → "여전히 공통 유지가 합리적인지" 코멘트 남기기
- [ ] 비대화 징후(옵션 분기 증가)? → 분리/헤드리스 검토
- [ ] 안 쓰이는 공통? → deprecated + 대체 제시

> 가장 현실적인 해결책 = 공통 컴포넌트에 대해 **더 자주·더 솔직하게 질문하기.**
