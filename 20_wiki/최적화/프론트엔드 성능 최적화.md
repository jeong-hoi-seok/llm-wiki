# 프론트엔드 성능 최적화

## 한 줄 요약
프론트엔드 성능 최적화는 사용자가 화면을 빠르게 보고, 빠르게 조작하고, 적은 데이터로 안정적으로 사용할 수 있도록 **리소스 크기·렌더링 차단·이미지/폰트·JavaScript 실행 비용·캐싱**을 관리하는 작업이다.

## 핵심 원칙
성능은 배포 직전 압축으로 해결하는 문제가 아니다. 설계·구현·리뷰·배포·모니터링 전 과정에서 관리해야 한다.

우선순위는 다음 순서로 둔다.

1. **측정**: Lighthouse, PageSpeed Insights, WebPageTest, DevTools, bundle analyzer로 병목을 확인한다.
2. **예산**: 페이지 무게, JavaScript bundle size, 이미지 용량, 폰트 용량에 상한을 둔다.
3. **전송 최적화**: 압축, 캐싱, CDN, HTTP cache header, Brotli/Gzip을 적용한다.
4. **렌더링 최적화**: CSS/JS가 초기 렌더링을 막지 않게 한다.
5. **리소스 최적화**: 이미지·폰트·외부 스크립트를 필요한 만큼만 보낸다.
6. **런타임 최적화**: JavaScript 실행 비용과 dependency 크기를 관리한다.

## 빠른 체크리스트

### HTML
- [ ] 운영 HTML은 minify한다.
- [ ] CSS는 JavaScript보다 먼저 로드한다.
- [ ] iframe은 꼭 필요한 경우에만 사용한다.
- [ ] 필요한 외부 origin에는 `preconnect`, `dns-prefetch`, `preload` 같은 resource hint를 검토한다.

### CSS
- [ ] 운영 CSS는 minify한다.
- [ ] 렌더링을 막는 CSS를 줄인다.
- [ ] critical CSS는 초기 화면에 필요한 만큼만 인라인한다.
- [ ] 사용하지 않는 CSS selector를 제거한다.
- [ ] CSS selector 복잡도와 중복을 주기적으로 점검한다.
- [ ] CSS concatenation은 HTTP/1 환경에서만 우선 고려하고, HTTP/2 이상에서는 실제 측정 후 판단한다.

### Fonts
- [ ] WOFF2를 우선 사용한다.
- [ ] 외부 폰트 서버에는 `preconnect`를 검토한다.
- [ ] 모든 font variant를 합친 용량이 과하지 않은지 확인한다.
- [ ] `font-display`로 invisible text를 줄인다.
- [ ] 브랜드상 꼭 필요한 굵기·스타일만 로드한다.

### Images
- [ ] 이미지는 압축하고 적절한 품질로 저장한다.
- [ ] 사진, 아이콘, 일러스트 성격에 맞는 포맷을 선택한다.
- [ ] 가능하면 SVG 같은 vector를 사용한다.
- [ ] `<img>`에 `width`, `height`를 지정해 layout shift를 줄인다.
- [ ] offscreen 이미지는 lazy loading한다.
- [ ] `srcset`, `picture`로 화면 크기에 맞는 이미지를 제공한다.
- [ ] 아주 작은 이미지 외에는 Base64 인라인을 피한다.

### JavaScript
- [ ] 운영 JS는 minify한다.
- [ ] `<script>`는 `defer` 또는 필요한 경우 `async`로 로드한다.
- [ ] body 중간에 여러 inline script를 흩뿌리지 않는다.
- [ ] dependency 크기를 확인하고 무거운 라이브러리를 대체 가능한지 본다.
- [ ] bundle analyzer로 큰 모듈과 중복 의존성을 확인한다.
- [ ] runtime 성능은 Chrome DevTools Performance/Memory로 profiling한다.
- [ ] PWA라면 Service Worker 캐싱 전략을 검토한다.

### Server / Delivery
- [ ] HTTPS를 사용한다.
- [ ] page weight를 관리한다. 가능하면 500KB에 가깝게, 최소한 1500KB 이하를 목표로 본다.
- [ ] 초기 page load를 3초 이내로 줄이는 것을 목표로 한다.
- [ ] TTFB가 과하게 길지 않은지 확인한다.
- [ ] 불필요한 HTTP request를 줄인다.
- [ ] asset은 CDN 사용을 검토한다.
- [ ] 404 asset 요청이 없는지 확인한다.
- [ ] HTTP cache header를 명시한다.
- [ ] Gzip 또는 Brotli 압축을 적용한다.
- [ ] cookie 크기와 개수를 관리한다.

## 실무 적용 기준

### 새 기능 PR에서 확인할 것
- 새 dependency를 추가했는가? 크기와 대체재를 확인한다.
- 새 이미지가 들어갔는가? 압축, 크기, `width/height`, lazy loading을 확인한다.
- 새 폰트/variant가 들어갔는가? 꼭 필요한지 확인한다.
- 새 페이지가 생겼는가? Lighthouse와 WebPageTest 기준으로 초기 로딩을 확인한다.
- 새 CSS가 많아졌는가? 미사용 selector와 중복을 확인한다.

### 성능 예산이 필요한 경우
다음 신호가 있으면 성능 예산을 문서화한다.

- 페이지가 늘어날수록 JS bundle이 계속 커진다.
- 이미지가 콘텐츠보다 먼저 성능 병목이 된다.
- 폰트 variant가 누적된다.
- Lighthouse 점수가 PR마다 흔들린다.
- 모바일 네트워크에서 첫 화면이 늦다.

예산 예시:

| 항목 | 기준 |
|---|---|
| 전체 page weight | 가능하면 500KB 근접, 최소 1500KB 이하 목표 |
| 이미지 | 화면에 필요한 실제 크기에 맞게 제공 |
| 폰트 | 필요한 family/weight/style만 로드 |
| JavaScript | route 단위로 분리하고 큰 dependency 추가 시 리뷰 |
| HTTP request | 불필요한 외부 script, iframe, 추적 스크립트 제한 |

## Next.js 프로젝트에서의 적용
[[Next.js]]를 쓰면 프레임워크 기능을 먼저 활용한다.

- 이미지: `next/image`의 sizing, lazy loading, format optimization을 검토한다.
- 폰트: `next/font`로 self-hosting과 layout shift 감소를 검토한다.
- 라우트: route 단위 [[코드 스플리팅]]을 전제로 화면을 설계한다.
- 서버 컴포넌트: 상호작용이 필요 없는 UI는 client bundle에 넣지 않는다.
- third-party script: 필요한 시점과 strategy를 명확히 한다.

## 도구
| 목적 | 도구 |
|---|---|
| 전체 성능 진단 | Lighthouse, PageSpeed Insights, WebPageTest |
| 네트워크·렌더링 분석 | Chrome DevTools Network/Performance |
| 번들 크기 분석 | bundle analyzer, Bundlephobia |
| CSS 미사용 확인 | Chrome DevTools Coverage |
| 이미지 점검 | Lighthouse, Cloudinary 분석 도구, SVGOMG |
| 지속 모니터링 | SpeedCurve, DebugBear, Calibre 같은 성능 모니터링 도구 |

## 에이전트 적용 기준
- 성능 문서를 만들거나 리뷰할 때는 “체크리스트 나열”보다 병목 우선순위를 먼저 잡는다.
- 새 프로젝트 문서에는 성능 예산을 프로젝트 성격에 맞게 구체 숫자로 남긴다.
- UI 구현 중 이미지·폰트·외부 script가 추가되면 성능 영향도 함께 기록한다.
- 성능 최적화는 [[가독성]]과 충돌할 수 있다. 복잡한 최적화는 실제 측정 결과가 있을 때 적용한다.
- 특정 프레임워크 문서보다 프로젝트 `docs/`의 배포 환경과 측정 기준을 우선한다.
