# 디버깅

## 한 줄 요약
프로그램이 예상대로 동작하지 않을 때 원인을 찾아 해결하는 과정. **감(感)이 아니라 체계적으로** — 진단 → 수정 → 예방 루프. (토스 디버깅 Fundamentals) 상위: [[index]]

## 3단계 루프

### 1. 🔍 진단 (Diagnose)
- 에러 메시지 전체를 구글에 붙여넣기 전에, **원인을 구체적으로 좁히기**.
- 에러 메시지 분석, 호출/의존 맵으로 문제 범위 특정. (raw: `pages/diagnose/`)

### 2. 🔧 수정 (Fix)
증상이 아니라 **근본 원인**을 고친다.
- **근본 원인 수정** — 겉보기 해결(임시방편)은 재발·파급. 예: `find`가 `undefined` 반환 → 옵셔널 체이닝/기본값으로 원인 처리. (raw: `pages/fix/correct.md`)
- **데드코드 제거** — 미사용 유틸·CSS·끝난 A/B 실험 코드는 디버깅 방해. 기능 제거 시점에 함께 삭제. (raw: `pages/fix/dead-code.md`) → [[결합도]]·[[가독성]]
- **순수 함수 분리** — 비즈니스 로직을 UI에서 떼어 순수 함수로 → 독립 테스트·재사용↑. (raw: `pages/fix/pure.md`) → [[응집도]]

### 3. 🛡️ 예방 (Prevent)
버그 수정은 끝이 아니라 시스템·팀 개선의 출발점.
- **에러 로그 상세** — 사용자 상태(버전·기기·OS)·재현 조건 기록 → 원인 분석·재발 방지. (raw: `pages/prevent/error-log.md`)
- **버그 리포트** — 문제 상황·원인·해결을 기록(위키·PR 링크). 재현 환경을 구체적으로. (raw: `pages/prevent/bug-report.md`)
- 팀 공유로 같은 실수 반복 방지.
