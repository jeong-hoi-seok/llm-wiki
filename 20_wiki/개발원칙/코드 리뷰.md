# 코드 리뷰

## 한 줄 요약
변경 코드를 머지 전에 검토해 버그·설계 문제·테스트 공백을 거르는 과정. 전체를 균일하게 보는 게 아니라 **변경의 영향 범위(blast radius)와 위험도 순으로** 본다. AI 리뷰는 변경+영향 노드만 컨텍스트에 넣어 토큰을 아낀다. (출처: `code-review-graph` → 도구 상세는 [[code-review-graph 요약]]) 상위: [[index]] · 관련: [[MR PR 작성 가이드]] · [[디버깅]]

## 무엇을 보나

리뷰 항목은 변경 단위마다 다음을 확인한다.

- **정확성·버그** — 변경 로직이 의도대로 동작하는가, 엣지 케이스 누락은 없는가.
- **영향 범위** — 이 변경에 의존하는 호출자·의존 대상·테스트가 무엇인가.
- **테스트 커버리지** — 변경된 함수에 테스트가 있는가. 없으면 구체적 테스트 케이스를 제안. → [[단위 테스트]]
- **공개 API 변경** — 시그니처·동작이 바뀌면 호출자가 전부 갱신됐는가 (breaking change).
- **스타일·가독성** — 의미 변화 없는 정리. 비중은 위 항목보다 낮다. → [[가독성]]

## 영향 범위(Blast Radius) 기반 리뷰

핵심 아이디어: 변경 파일만 보지 말고 **변경이 전파되는 범위**를 본다.

- 파일이 변경되면 그 코드를 부르는 **호출자(callers)**, 그 코드가 의존하는 **의존 대상(dependencies)**, 관련 **테스트**가 영향 범위다.
- 호출자가 바뀐 함수: 시그니처·동작이 그대로 맞는지 검증.
- 상속 구조 변경: 하위 타입 치환 가능성(Liskov) 확인.
- 의존하는 다른 코드가 많은 파일: 고위험 변경으로 분류.

[추론] 도구 없이도 동일 원칙 적용 가능 — `git diff`로 변경 함수를 찾고, 에디터의 "참조 찾기(find references)"로 호출자를 따라가며 영향 범위를 수동으로 좁힌다.

## 위험도 기반 우선순위

리뷰 결과는 위험도(high / medium / low)로 묶어 보고한다.

- **High**: 의존 대상이 많은 코드, 공개 API 변경, 테스트 없는 변경.
- 큰 PR은 **영향이 가장 큰 파일(의존자가 많은 것)부터** 본다. 시간이 한정될 때 우선순위가 분명해진다.
- 변경 함수에 테스트가 없으면 자동으로 플래그하고 케이스를 제안.

## 리뷰 결과 구조화 출력

리뷰는 자유 서술이 아니라 정해진 틀로 남긴다. `code-review-graph`의 PR 리뷰 출력 형식:

```md
## PR Review: <제목>

### Summary
<1~3문장 개요>

### Risk Assessment
- Overall risk: Low / Medium / High
- Blast radius: X개 파일, Y개 함수 영향
- Test coverage: 변경 함수 N개 중 M개 커버

### File-by-File Review
#### <파일 경로>
- Changes: <설명>
- Impact: <누가 의존하는가>
- Issues: <버그·스타일·우려>

### Missing Tests
- <파일>의 <함수> — 테스트 없음

### Recommendations
1. <실행 가능한 제안>
```

## 리뷰 워크플로 3종

변경 범위에 따라 리뷰 단위를 고른다.

| 워크플로 | 대상 | 특징 |
|---|---|---|
| **PR 전체 리뷰** (`review-pr`) | `git diff main...<branch>` 전체 | 모든 커밋 변경 + 영향 범위 종합. 고위험 파일 우선. |
| **델타 리뷰** (`review-delta`) | 마지막 커밋 이후 변경 | 변경 노드 + 2-hop 이웃만. 토큰 5~10배 절감. |
| **변경 리뷰** (`review-changes`) | 위험 점수 기반 변경 | 위험도별 그룹핑 + 영향 실행 흐름 + 머지 권고. |

## AI 코드 리뷰의 토큰 효율

AI 코딩 도구는 리뷰 때 코드베이스 큰 부분을 반복해 읽어 토큰을 낭비한다. 이를 줄이는 원칙:

- **변경 + 영향 노드만** 컨텍스트에 넣는다. 전체 파일은 명시 요청 없으면 포함하지 않는다.
- 최소 컨텍스트를 먼저 조회하고(`get_minimal_context`), 부족할 때만 상세도를 올린다.
- `code-review-graph`는 코드를 Tree-sitter로 AST 그래프(노드=함수·클래스·import, 엣지=호출·상속·테스트)로 만들어 SQLite에 저장하고, 리뷰 시점에 쿼리해 읽어야 할 **최소 파일 집합**만 계산한다. 실제 저장소 기준 평균 8.2배(최대 27배) 토큰 절감. [검증 필요] 벤치마크 수치는 도구 자체 평가 결과.

[추론] 그래프 도구가 없어도 "변경과 그 영향 범위만 본다"는 원칙은 사람·AI 리뷰 모두에 적용된다. 모노레포일수록 효과가 크다 — 무관 파일을 리뷰 컨텍스트에서 제외하는 것이 핵심.
