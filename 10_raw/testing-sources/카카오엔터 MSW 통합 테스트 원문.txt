“FE 테스트의 든든한 조력자 MSW”

최근에는 FE에도 jest, cypress와 같은 훌륭한 테스트 도구들이 등장하면서, 효율적으로 통합 테스트나 e2e 테스트를 작성할 수 있게 되었습니다. 하지만 테스트를 작성하다 보면 더미 데이터를 만들거나 많은 API를 모킹해가며 테스트를 진행해야 하는 것은 변함이 없었습니다. 그리고 이것은 꽤나 많은 비용을 발생시키는 작업이었습니다.

오늘은 이런 문제를 해결하기 위해 MSW(Mock Service Worker)라는 도구를 통합테스트에 도입하게 된 경험과 테스트에서 어떻게 활용할 수 있는지 소개해 드리려 합니다.

해당 글은 react(typescript), jest(testing-library)를 기반하여 작성되었음을 알려드립니다.

MSW의 도입
들어가기에 앞서 Front-End에서 작성하고 있는 보편적인 테스트들을 알아보겠습니다.
대표적으로 유틸 함수를 테스트하거나 작은 단위의 컴포넌트를 테스트하는 코드들이 있을 것입니다.

테스트 코드 예시

많은 개발자들이 이러한 단위 테스트를 작성함으로써 각 컴포넌트의 무결성을 보장하려 하는데요, 위와 같이 작은 단위의 테스트는 복잡도가 낮아 정상적으로 테스트가 가능합니다.
하지만 통합 테스트를 넘어가면서 문제가 발생하기 시작합니다. 여러 컴포넌트가 모여 하나의 기능을 수행하려면 서버와 통신해야 하는 경우가 많기 때문입니다.

검색어 입력과 검색 결과로 구성된 검색 페이지의 사례를 살펴보겠습니다.

간단한 검색페이지 구성

// Search.tsx

/**
 * 검색어 입력과 검색 결과로 구성된 검색 페이지
 * fetchData: API를 호출해서 데이터를 갱신하는 함수
 * SearchInput : 검색어 입력 컴포넌트
 * SearchResultList : 검색 결과 리스트
 */ 

export default function Search() {
  const [data, setData] = useState<ResultData>([]);

  const fetchData = async (keyword: string) => {
    const response: ResponseType = await axios({
      url: '/search',
      method: 'get',
      data: { keyword },
    });
    setData(response.data.data);
  };

  return (
    <div>
      <SearchInput onPress={fetchData} />
      <SearchResultList data={data} />
    </div>
  );
}
검색페이지 컴포넌트


이때 검색어 입력 컴포넌트에 특정 키워드를 입력했을 때 검색 결과가 잘 보이는지 테스트하려면 검색어 입력 컴포넌트에서 API 호출을 통해 서버로부터 데이터를 가져와야 합니다.

하지만 만약 API가 아직 개발 중이라면 어떻게 될까요? 혹은 원하는 데이터를 넘겨주지 않는다면요?

다음의 작업을 수행하는 테스트 코드를 통해 어떤 결과가 나오는지 확인해 보도록 합시다.

검색어 입력 컴포넌트에 특정 키워드를 입력합니다.
입력한 검색어 키워드를 사용하여 API 호출을 하고 데이터를 가져옵니다.
가져온 데이터를 검색 결과 컴포넌트에 반영하여 렌더링 합니다.
describe('검색페이지 통합테스트', () => {
  it('검색어 입력후 엔터 키를 입력하면 검색된 앨범을 보여준다.', async () => {
    render(<Search />);

    const inputBox = screen.getByRole('textbox');

    //검색어 입력
    fireEvent.change(inputBox, { target: { value: '아이유' } });
    //API 호출
    fireEvent.keyDown(inputBox, { key: 'Enter' });

    await waitFor(() => {
      //렌더링 결과 확인
      expect(screen.queryByText('조각집')).toBeInTheDocument();
    });
  });
});

테스트코드 실행시 에러 발생

예상한 대로 에러가 발생하게 됩니다.

결국 해당 테스트를 건너뛰게 되고 “검색을 통해 검색 결과를 가져올 수 있다” 라는 기능을 보장할 수 없게 됩니다.
설령 나중에 추가하게 된다고 해도 핵심 기능의 치명적인 이슈를 늦게 발견하게 되어 많은 코드를 고쳐야 할 수도 있습니다.

따라서 저희는 이러한 문제를 해결하고 사전에 방지하고자 MSW라는 도구를 도입하여 통합 테스트를 진행하고 있습니다.
MSW는 API 요청을 가로채서 사전에 설정해둔 목업 데이터를 넘겨주도록 설정해 주는 도구입니다.
서버 개발자와 협의가 이뤄진 API 스펙만 있다면 미리 목업을 만들어, 실제 데이터를 호출하듯이 테스트를 해볼 수 있게 되는 것입니다.
많은 블로그에서 MSW를 소개하는 내용을 다루고 있기 때문에 자세한 설명은 생략하고 어떻게 활용할 수 있는지 바로 알아보도록 하겠습니다.

MSW를 활용해서 진행할 수 있는 대표적인 테스트들을 살펴보도록 하겠습니다.
다음과 같은 테스트를 통과하는 코드를 목표로 할 것입니다.

서버에서 데이터를 정상적으로 잘 가져와서 렌더링 하는가

1. 목업을 통해 데이터 가져오기
우선 src/mocks 폴더 아래에 api 폴더를 생성하고 해당 경로에 data 폴더를 생성하도록 하겠습니다. 그리고 searchResult.ts 파일과 searchResultData.ts 파일을 다음과 같은 경로로 생성해 줍니다.

파일명과 폴더구조는 본인의 컨벤션에 맞추면 됩니다.

src
├── mocks
│   ├── api
│   │   ├── data
│   │   │   └── searchResultData.ts
│   │   └── searchResult.ts
│   ├── handlers.ts
│   └── server.ts
src/mocks/api/data : API 호출 시 응답할 데이터를 모아두는 폴더
src/mocks/api : 모킹할 API를 모아두는 폴더

그 후 각 파일에 다음과 같이 설정해 주도록 합시다.

// handlers.ts
/**
 * 모킹하려하는 API를 가져와 handlers에 넣어줍니다.
 * handler를 따로 분리하는 이유는 연관된 API끼리 묶어주기 위함입니다.
 */
import searchResultHandler from './api/searchResult';

const handlers = [...searchResultHandler];

export default handlers;
// searchResult.ts
/**
 * 모킹하려는 API를 설정합니다.
 * 첫 번째 인자로 경로, 두 번째 인자로 request, response, context를 파라미터로 넘겨받는 콜백 함수를 넣어줍니다.
 * response와 context.json을 아래와 같이 조합하면 API 응답 값을 넣을 수 있습니다.
 * 원하는 API가 여러 개라면 handlers 배열에 여러 개를 넣어주면 됩니다.
 */
import { rest } from 'msw';

import { searchResultData } from 'mocks/api/data/searchResultData';

const handlers = [
  rest.get('/search', (_, res, ctx) => res(ctx.json(searchResultData))),
];

export default handlers;
// searchResultData.ts
/**
 * API 호출 시 응답받을 데이터를 설정합니다.
 */
export const searchResultData = {
  data: [
    {
      image: 'https://...생략',
      title: '조각집',
      type: 'rect',
    },
  ],
};
이제 MSW를 활용해 목업 데이터를 가져오는 기본적인 세팅이 완료되었습니다.
아래 테스트를 다시 실행해서 모킹 된 데이터를 잘 가져오는지 결과를 확인해 봅시다.

2. 동적으로 데이터 변경하기
다음은 런타임에 동적으로 응답 데이터를 변경하는 테스트 코드를 작성해 보도록 하겠습니다.
만약에 새로 작성해야 하는 테스트에 다른 응답 데이터가 필요하면 어떻게 해야 할까요?

search API를 호출했는데 응답이 rect(정사각 모양 앨범)일 수도 있고, rounded(둥근 앨범)일 수도 있을 것입니다.

데이터를 바꿔가면서 테스트 코드를 여러 번 돌려야 할까요?
다행히 MSW는 런타임에 동적으로 특정 API의 모킹 데이터를 변경할 수 있습니다.
테스트 코드에 아래와 같은 전처리 작업을 추가합니다.

// Search.test.tsx
...
import { rest } from "msw";
import { server } from 'mocks/server';
import { searchResultData } from 'mocks/api/data/searchResultData';

describe('검색페이지 통합테스트', () => {
  it('type이 rounded인 경우 둥근 앨범을 보여준다.', async () => {
    /**
     * 타입을 변경해서 응답 데이터를 등록합니다
     * 혹은 목업 데이터 파일을 미리 만들어 두고 해당 데이터를 등록하는 것도 좋은 방법입니다.
     */
    const result = searchResultData;
    result.data = result.data.map(item => ({ ...item, type : 'rounded' }));
    server.use(rest.get('/search', (_, res, ctx) => res(ctx.json(result))));

    render(<Search />);

    const inputBox = screen.getByRole('textbox');
    fireEvent.change(inputBox, { target: { value: '아이유' } });
    fireEvent.keyDown(inputBox, { key: 'Enter' });

    await waitFor(() => {
      expect(screen.queryByTestId('card-image')).toHaveClass(
        'card-rounded-border',
      );
    });
  });
});
위의 예시는 데이터의 결과에 따른 컴포넌트 조건부 렌더링을 해주는 모습입니다.
여기서 우리는 server.use를 통해 모킹 데이터를 동적으로 바꿀 수 있음에 주목하고 여러 가지 테스트를 할 수 있습니다.

4. 에러 발생 상황 테스트하기
만약에 어느 날 갑자기 API에 문제가 발생하면 어떻게 될까요? API를 호출 시 에러가 발생하게 되고 에러 핸들링이 잘 되어있다면 fallback을 실행하고 그렇지 않다면 페이지가 다운되어 버릴 것입니다.
그러면 이런 네트워크 에러를 어떻게 발생시킬 수 있을까요? 이번엔 데이터파일이 아닌 목업 설정을 통해 구현할 수 있습니다. 가장 흔하게 발생하는 404 에러를 발생시켜 보도록 하겠습니다.

function overrideSearchResultWithErrorData() {
  server.use(
    rest.get('/search', (_, res, ctx) =>
      res(ctx.status(404), ctx.json({ errorMessage: 'Not Found' })),
    ),
  );
}

it('네트워크 에러가 발생하면 안내문을 출력한다.', async () => {
  overrideSearchResultWithErrorData();
  render(<Search />);

  const inputBox = screen.getByRole('textbox');
  fireEvent.change(inputBox, { target: { value: '아이유' } });
  fireEvent.keyDown(inputBox, { key: 'Enter' });

  await waitFor(() => {
    expect(screen.queryByText('검색 결과가 없습니다.')).toBeInTheDocument();
  });
});
그 후 데이터를 가져오는 부분에 try catch를 통해 에러 상황에 맞는 데이터를 넣어주도록 수정합니다.

// Search.tsx
const fetchData = async (keyword: string) => {
  try {
    const response: ResponseType = await axios({
      url: '/search',
      method: 'get',
      data: { keyword },
    });
    setData(response.data.data);
  } catch ({ response: { status } }) {
    // 서버와 협의한 내용에 맞춰 에러 핸들링
    if (status === 404) {
      setData([]);
    } else if (status === 403) {
      ...
    }
  }
};

5. 결과 확인하기

picture6

통합테스트 통과!

앞서 목표로 했던 데이터 패칭, 조건부 렌더링, 엣지 케이스, 에러 테스트를 통과했습니다. 물론 실무에서는 이것보다 테스트 케이스가 훨씬 많고 복잡할 것입니다. 사실 위의 테스트의 경우 API 결과에 따른 컴포넌트 하나의 변화만 검증했기 때문에 어떻게 보면 단위 테스트로도 충분히 커버할 수 있는 영역입니다. 하지만 특정 API 결과에 따라 여러 개의 컴포넌트가 변경되고, 케이스가 복잡하다면 위와 같은 통합 테스트 코드가 강력한 역할을 수행할 것입니다.


그 외 MSW 활용 방법
스토리북 연동
요즘 스토리북을 통해 시각적으로 테스트를 진행하는 경우가 많습니다. 다양한 시나리오에 해당하는 렌더링 결과를 확인하려면 당연하게 MSW가 필수적으로 필요할 것입니다. 이에 대해서는 이전에 카카오 엔터테인먼트 리치가 소개해 주신 글(MSW 모킹 코드 재사용하기 feat. Storybook, Jest)이 있으므로 해당 글을 참고하시면 많은 도움이 되실 것 같습니다.

개발 서버 연동
Front-End 개발을 하다 보면 테스트 코드만 보면서 개발을 할 수는 없을 것입니다. 개발 서버를 열어 확인하는 과정이 필요하며 그러기 위해서는 추가적인 설정이 필요합니다. 공식 홈페이지 브라우저 환경 세팅에서 소개해 준 방식으로 간단하게 서비스워커를 설치하고 테스트해 볼 수 있습니다.

맺음말
통합 테스트를 진행하면서 발생하는 API와 관련된 이슈는 애자일과 같은 빠른 개발 방법론을 추구하는 오늘날의 많은 프로젝트에서 마주하는 문제입니다. 따라서 정해진 시점까지 프로덕트 혹은 mvp 모델을 출시하기 위해, API 개발이 끝나지 않아도 Front-End 개발을 시작하는 병렬적인 작업을 하게 되는 경우가 발생하게 되고 이에 따라 위와 같은 문제점이 발생하게 됩니다.



picture7
프로젝트 이상과 현실의 불일치

과거에는 이런 문제를 해결하기 위해 주석을 활용하거나, 모킹서버를 직접 개발해서 우회시키는 등 비효율적이고 무결성을 보장할 수 없는 작업을 했었습니다. 하지만 이제는 MSW를 활용하여 거의 실제와 유사한 환경에서 통합 테스트를 효율적으로 진행할 수 있게 되었습니다. 테스트를 하기 위한 별도의 처리를 하지 않아도 되기 때문에 테스트를 통과한다면 현실에서도 동일한 상황에 알맞게 대처하는 것을 보장하게 됩니다. 또 다음 스프린트가 진행됨에 따라 변경된 API 스펙을 미리 적용하여 기존 코드와의 정합성을 빠르게 체크할 수도 있어 유지 보수성도 훨씬 증진됨을 경험하게 될 것입니다. 그러니 MSW를 도입해 환경을 구축하는 것은 이제 선택이 아닌 필수라고 생각합니다.

이상 MSW를 활용하여 효율적인 통합 테스트를 작성함에 도움이 되기를 바라며 글을 마칩니다. 감사합니다.
