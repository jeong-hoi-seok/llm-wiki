# 공통 컴포넌트, 잘 만들고 있나요? (카카오페이손해보험 FE)

등장인물: 입사 4개월차 차차(chacha), 3년차 페페(pepe).

## 시작하며 — 공통 컴포넌트가 이렇게 많다니
"이거 공통으로 빼야 할까?" 누구나 마주하는 질문. 차차가 레포 열었을 때 공통 컴포넌트가 매우 많음. 재료 많은 건 장점이지만, 어디까지가 사용 가치 있는 컴포넌트인지 알기 어려움.
- chacha: 공통 많으니 어떤 기준으로 쓸지 막막. docs 있었으면 구조 이해 수월했을 것.
- pepe: 지금 구조는 처음부터 이런 게 아님. 상품 늘며 재사용 위해 조금씩 공통화, 그 과정에서 세밀해짐.
공통 증가 = 의도된 선택이지만, 문서화 안 된 결정이 쌓여 히스토리 공유 어려운 상태. 이 글은 정답 제시가 아니라, 공통 컴포넌트가 생산성을 높이다 언제부터 갉아먹는지에 대한 기록.

## 공통 컴포넌트의 탄생 — 언제 만드나
차차: 새 TextArea 필요 → 이미 있는 StaticLabelTextArea 발견. 겉보기엔 공통화 충분해 보임. 비슷한 입력 컴포넌트 여러 화면 반복 가능성.
하지만 구현부 보니 걸림:
- 내부적으로 **deprecated 예정 컴포넌트에 의존**
- 실제 **사용처 2곳뿐**
→ 공통화 이점 < 단점. 향후 변경 시 유지 비용 더 클 신호. 특히 의존하는 하위 컴포넌트 변경 시 영향 범위 예측 어려움이 가장 큰 부담.
질문이 '공통 만들 수 있나'에서 '언제 잘 만들었다 말할 수 있나'로 확장.

## 공통 컴포넌트의 사춘기 — 비대화
명확한 경계 없이 역할 계속 받으면 본래 목적과 다른 모습이 됨.
예: 모바일 채널 CoverageSelectCard. 보험 상품 선택이라는 단순 UI로 시작했으나 다양한 요구사항 흡수. 특히 바텀시트 관련 동작이 내부로 모임:
- 바텀시트 열기/닫기
- iOS 이슈 대응 body height 조작
- 드래그 인터랙션
- 애니메이션·딜레이 제어
- pepe: 바텀시트 문제 해결 위해 로직 쌓임 → 점점 많은 책임 동시 수행. 새 옵션 추가 때마다 컴포넌트 자체 수정.
성장 과정: 작고 단순 시작 → 요구사항마다 기능 덧붙임 → 리팩토링 시점 못 잡음 → 여러 역할 동시 수행.
보험 도메인의 복잡한 비즈니스 규칙·예외 케이스도 비대화에 영향.
질문은 "이 컴포넌트가 나쁜가?"가 아니라 "역할들을 언제·어떤 기준으로 나누는 게 유지보수에 적합한가?"

## 아직 독립 준비 안 된 컴포넌트
모두 비대화하는 건 아님. 어떤 건 충분히 성장 전에 '공통' 이름을 먼저 가짐.
예: DelayRender/RenderAfter — delayTime 전엔 fallback, 후엔 children 렌더. 코드 작고 도메인 무관 순수 UI 유틸. 공통 분리는 당시 합리적. 단:
- "정말 공통으로 쓰이나?" — 소수 화면만 사용. 앱 내부 처리로 충분한 규모.
- 공통이라 보아도 틀리지 않지만 이상적 공통이라 하기도 어려운 경계선.
중요한 건 정답이 아니라 판단 기준을 팀 내 공유:
- 몇 번 반복부터 공통으로 볼 것인가
- 실제 사용 빈도 vs 관리 비용 저울질
- "지금은 공통 아니다" 말할 시점은 언제인가
DelayRender = '언제 만드나'보다 '언제까지 공통 유지하나' 고민하게 한 사례.

## 공통 컴포넌트의 결혼(결합) — 템플릿
chacha: 템플릿은 어떤 기준에서 만들어지나?
pepe: 보험 도메인은 템플릿 불가피한 순간 있음. 청약 씬(가입설계, 계약체결동의, 미리보기 문서, 계약자확인사항 등) 대부분 청약 플로우에서 반드시 거치는 영역.
- **단일 컴포넌트(자유)**: 하나의 책임만. 입력받고 정보 보여주고 상태 표현.
- **느슨한 조합(관계 시작)**: 여러 단일 조합 → 화면/기능. 최소한의 관계 생김.
- **템플릿(자유 아님)**: 단순히 묶은 게 아니라 어떤 결정을 코드로 고정하는 구조.

언제 결합/분리?
- ✅ 그린 라이트(결합 OK): 같은 조합이 복붙됨(청약사항확인씬 거의 동일). 조합의 주체가 명확.
- ❌ 레드 라이트(아직 템플릿 아님): 화면마다 예외 계속 생김. 불확실한 미래 걸고 뺀 경우 / 충분한 검증·케이스 안 거침.
즉 템플릿 = 재사용 도구가 아니라 **변경하지 않기로 합의한 결정 집합**.

## 공통 컴포넌트를 위한 추모 — 죽음(deprecated)
원피스 명대사: "사람은 언제 죽는가? ... 사람들에게 잊혀졌을 때다!"
공통 컴포넌트도 죽음:
- 아무도 찾지 않을 때
- 외부 정책 요인(ex payfit 적용)
- 로직 너무 복잡해서 건들면 다 죽을 것 같을 때
유기된 컴포넌트는 재사용 이점보다 수정·검증 부담의 비용 요소가 됨. 이때 deprecated = 실패 선언이 아니라 비용 절감 선택.
deprecated 시작 = '사용 안 함'이 아니라 '의도적으로 선택되지 않음'. 새 요구 못 만족하거나 팀 기본 선택지에서 빠지는 순간 자연스럽게 죽음.
다수 조직이 공통 컴포넌트를 영구 자산으로 안 봄. 디자인 시스템 진화·조직 변화·유지비용 증가로 의도적 deprecated + 대체 제시. (예: Shopify Polaris, 자체 디자인시스템 KID 종료)

## 마치며 — 해결책: 헤드리스 컴포넌트
chacha: 가이드도 지키고 자유도도 갖는 방안 없나?
pepe: **헤드리스 컴포넌트(Headless Component)**.
- unstyled를 넘어 로직 기반 재사용성/확장성/UI 커스터마이징 자유 제공.
- 상태관리·인터랙션 로직·접근성 처리 같은 **근본 동작만 공통 제공**, 시각 표현은 사용처가 결정하도록 위임.
- 대표 라이브러리: Reach UI(reach.tech) — 모든 컴포넌트가 로직만, 스타일 제외.
접근성 직접 구현은 브라우저별 차이·키보드 포커스·ARIA 등 공수 큼 → 매번 구현 비현실적.
제안 구조: Headless(로직+접근성)를 베이스로 → 가이드 맞춘 기본 UI 버전 + 서비스 특성 맞춘 자유 UI 버전.
디자인 완전 균일화가 목표 아니라면, 안정성·접근성은 공통, 디자인은 유연 허용 구조가 유지보수·확장성의 돌파구.

## 팀이 던지기로 한 질문
공통 컴포넌트 만들기 전:
- 이미 존재하는 컴포넌트 조합으로 해결 못 하나
- 지금은 공통 아니라고 해도 괜찮은 시점 아닌가
- 나중에 공통으로 옮겨도 문제없는 구조인가
기존 공통 사용 시에도 PR·리뷰에서 "여전히 공통 유지가 합리적인지" 질문 남기기.
가장 현실적 해결책 = 공통 컴포넌트에 대해 더 자주·솔직하게 질문하는 것.
