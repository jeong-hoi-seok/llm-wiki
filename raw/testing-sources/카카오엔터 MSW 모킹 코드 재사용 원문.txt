마치며
필자가 사용하고 있는 MSW는 Mock Service Worker의 약자이며 API를 모킹할 때 사용하는 도구이다. 리소스 요청을 가로채서 수정할 수 있는 Service Worker의 기능을 사용하여 모킹할 수 있다. Storybook으로 시각적 테스트를 하며 개발할 때 API 모킹이 필요할 때가 있다. API의 다양한 응답에 따라 달라져야 하는 컴포넌트를 시각적으로 테스트할 때 MSW가 매우 유용하게 사용된다. Storybook뿐만 아니라 브라우저에서 구동되는 페이지에는 대부분 사용할 수 있으며 Node.js도 지원되므로 활용 범위가 넓다.

이 글은 MSW로 모킹한 스토리 컴포넌트를 Jest + Testing-library에서 재사용하는 아이디어를 제시한다. 각 도구에 대해서 자세한 내용은 다루지 않으며 개념상 간단히 언급만 할 것이다. 전체적인 개발 흐름을 요약하면 다음과 같다.

스토리북 스토리 작성
스토리에 MSW 모킹 코드 추가
자동화 혹은 통합 테스트
Jest에서 스토리 Import
스토리에 추가된 MSW 모킹 재사용
다음은 사용된 도구 목록이다.

MSW - API 모킹
Storybook - 컴포넌트별로 시각적 테스트를 할 수 있는 도구

Jest, testing-library - 테스팅 도구

이 방식은 다음의 요건을 만족해야 한다.

모킹을 한 군데서 하고 여러 곳에서 사용한다. Storybook, Jest
테스트 시 API 모킹이 다른 테스트에 영향을 주지 않아야 한다.
이제부터 중복 코드 없이 하나의 모킹 코드로 스토리북과 Jest 테스트에서 재사용할 수 있는 방법을 살펴보자.

Storybook과 Jest에서 중복 모킹 상태
Storybook에서 MSW 사용하기
Storybook에서 MSW를 사용하기 위해서는 msw-storybook-addon을 설치해야 한다.

npm i -D msw-storybook-addon
그리고 .storybook/preview.js파일에 다음과 같이 초기화 루틴을 작성한다.

import { initialize, mswDecorator } from "msw-storybook-addon";

// MSW 초기화 함수 실행
initialize();

// MSW 데코레이터를 전역으로 적용
export const decorators = [mswDecorator];
이제 스토리에 MSW 모킹 코드를 추가해 보자. 필자는 멜론을 개발하고 있어서 앨범의 노래 목록을 표현하는 SampleAlbumSongList를 예제로 작성하였다. 예제 컴포넌트는 내부에서 API를 호출하며 응답에 따라 노래 목록을 표시한다. 다음 예제는 앨범에 노래가 없는 경우를 시각적으로 테스트하기 위하여 MSW가 가로채서 빈 객체로 응답한다. 그러면 앨범에 노래가 없는 경우 컴포넌트 상태를 확인할 수 있다.

코드 예제 1> 스토리북에서 MSW 모킹

import { rest } from "msw";

const Template = args => <SampleAlbumSongList {...args} />;

export const Album_Empty = Template.bind({});
Album_Empty.args = {
  albumId: "2433",
};
Album_Empty.parameters = {
  msw: {
    handlers: [
      rest.get("/api/album/song/list", (req, res, ctx) => {
        return res(ctx.status(200), ctx.json({}));
      }),
    ],
  },
};
Jest에서 MSW 사용하기
@storybook/testing-react는 스토리북에서 만든 컴포넌트를 테스트에서 재사용할 수 있도록 컴포넌트를 변환해 주는 도구이다. 스토리에서 데코레이터, args 등으로 시각적 테스트에 필요한 모킹 값들을 이미 설정했으므로 Jest에서도 재사용하는 것이 중복 코드를 줄이는 데 도움이 된다.

코드 예제 2> Jest + Testing-library에서 MSW 모킹

import { rest } from "msw";
import "@testing-library/jest-dom";
import { render, waitFor } from "@testing-library/react";
import { composeStories } from "@storybook/testing-react";
import * as SampleAlbumSongListStories from "./SampleAlbumSongList.stories";

// 테스트에서 사용할 수 있게 스토리를 변환한다.
const { Album_Empty } = composeStories(SampleAlbumSongListStories);

describe("SampleAlbumSongList", () => {
  it("앨범에 노래가 없는 경우 빈 리스트를 표시한다.", async () => {
    // 여기서 한번 더 MSW로 모킹해야 한다.
    server.use(
      rest.get("/api/album/song/list", (req, res, ctx) => {
        return res(ctx.status(200), ctx.json({}));
      })
    );

    const { container } = render(<Album_Empty />);
    await waitFor(() => expect(container).toHaveTextContent("Empty List"));
  });
});
무엇이 문제인가?
@storybook/testing-react을 사용함으로써 상당 부분 모킹 코드를 재사용할 수 있게 되었다. 하지만 MSW의 모킹 코드는 여전히 중복으로 정의된 것을 볼 수 있다. 중복으로 정의된 것은 모킹 코드를 별도의 모듈로 분리하고 import해서 사용하면 해결할 수 있다. 그러나 Jest에서 사용하려면 여전히 모킹 코드가 한번은 들어가야 한다. 테스트 코드를 읽을 때 부가정보들이 눈에 보이므로 깔끔하지 않다고 생각한다.

// handler.js
export const handler = rest.get("/api/album/song/list", (req, res, ctx) => {
  return res(ctx.status(200), ctx.json({}));
});

// test 파일
import { handler } from "./handler.js";
...

it("앨범에 노래가 없는 경우 빈 리스트를 표시한다.", async () => {
  // 여전히 모킹 코드를 작성해야 한다.
  server.use(handler);

  const { container } = render(<Album_Empty />);
  await waitFor(() => expect(container).toHaveTextContent("Empty List"));
});
테스트 코드에서 server.use(handler)와 같은 모킹 코드를 제거하고 가능한 깔끔하게 작성해 보자.

해결 방안
해결 방안을 고민할 때는 시간이 많이 들었지만 의외의 곳에서 답을 찾을 수가 있었다. msw-storybook-addon의 동작 방식을 분석하다가 힌트를 얻게 되었는데, 스토리에 MSW 모킹이 정의된 부분을 수행하는 코드였다. 전역 등록이 아닌 런타임에 use()로 적용한 모킹은 resetHandlers() 함수로 초기화할0 수 있다. use()를 사용하여스토리.parameters.msw로 모킹이 정의되었다면 기존 핸들러를 초기화(api.resetHandlers())하여 모킹의 사이드 이펙트를 없애고 새로운 모킹 핸들러를 추가(api.use(...msw)) 하는 것이다.

원본 코드 링크

export const mswDecorator: DecoratorFunction = (storyFn, context: DecoratorContext) => {
  const {
    parameters: { msw },
  } = context;

  if (api) {
    api.resetHandlers();

    if (msw) {
      if (Array.isArray(msw) && msw.length > 0) {
        // 배열 형태의 요청 핸들러 지원 (하위 호환).
        api.use(...msw);
      } else if ("handlers" in msw && msw.handlers) {
        // 원래 코드 주석 대신 타입을 표시하였다.
        /**
         * type MswParameter = {
         *   handlers: RequestHandler[] | Record<string, RequestHandler | RequestHandler[]>
         * }
         */
        const handlers = Object.values(msw.handlers)
          .filter(Boolean)
          .reduce((handlers, handlersList) => handlers.concat(handlersList), [] as RequestHandler[]);

        if (handlers.length > 0) {
          api.use(...handlers);
        }
      }
    }
  }

  return storyFn();
};
그럼 이 방식을 활용하여 어떻게 해결했는지 살펴보자.

@testing-library/react의 render 함수를 래핑하여 해결하기
Jest 테스트에서 위의 문제를 해결하기 위해 Custom Render를 활용하였다. Custom Render로 래핑을 하면 @testing-library/react의 render 함수 실행 시 원하는 API에 대해 모킹이 적용되도록 만들 수 있다. 코드를 보면 바로 이해가 될 텐데, Custom Render 함수가 호출되면 스토리.parameters.msw가 있는지 확인하고 모킹을 적용한다.

import { render } from "@testing-library/react";
import { setupServer } from "msw/node";

const server = setupServer();

function mockStoryHandlers(story) {
  server.use(...(story.type.parameters?.msw?.handlers || []));
}

const customRender = (ui, options) => {
  // 스토리에서 모킹된 것이 있다면 적용한다.
  mockStoryHandlers(ui);
  return render(ui, options);
};

export * from "@testing-library/react";
export { customRender as render };
간단한 래퍼 함수로 Jest에서 컴포넌트가 렌더링 될 때마다 server.use()를 사용해서 스토리의 모킹을 재사용했다. 이 방식을 사용하면 테스트 코드에서 모킹 관련 코드는 깔끔하게 제거될 수 있다. 테스트 코드의 가독성이 향상된 것을 느낄 수 있을 것이다.

// server.use()의 모킹 코드가 제거되었다.
it("앨범에 노래가 없는 경우 빈 리스트를 표시한다.", async () => {
  const { container } = render(<Album_Empty />);
  await waitFor(() => expect(container).toHaveTextContent("Empty List"));
});
또한 테스트별로 모킹에 의한 사이드 이펙트를 없애기 위해서 MSW의 가이드처럼 테스트 수행 후 모킹 핸들러를 초기화시켜준다.

import { server } from "shared/mocks/mswNodeServer";

// Establish API mocking before all tests.
beforeAll(() => server.listen());

// Reset any request handlers that we may add during the tests,
// so they don't affect other tests.
afterEach(() => server.resetHandlers());

// Clean up after the tests are finished.
afterAll(() => server.close());
트러블슈팅 - Node.js에서 MSW는 절대경로를 써야 한다.
일반적인 API를 호출할 때 url을 보통 호스트를 제외한 상대경로로 작성할 것이다. dev, sandbox, production 서버 등에서 구동될 때마다 따로 신경 쓰지 않기 위해서다. 그러나 MSW를 Node.js에서 구동할 때는 가이드에 따라 절대 경로만 써야 한다. 이렇게 되면 테스트 코드가 아닌 애플리케이션 코드에서도 절대 경로로 써야 MSW 모킹이 된다는 것인데 이는 매우 불편한 것으로 생각한다. 더군다나 url 문자열이 다르면 MSW는 모킹 매치가 되지 않는다.

코드에서 예시>

// 이렇게 상대 경로로 사용하고 싶을 것이다.
const query = useQuery(["/api/album/song/list", albumId], () => fetchSampleAlbum(albumId));

// 서버별로 url 문자열을 따로 구성해야 한다.
const query = useQuery(["https://dev.melon.com/api/album/song/list", albumId], () => fetchSampleAlbum(albumId));

// 불편하다.
const query = useQuery(["https://sandbox.melon.com/api/album/song/list", albumId], () => fetchSampleAlbum(albumId));
테스트 코드에서 예시

// 테스트 코드에서도 서버에 따라 url 문자열이 달라지므로 모킹을 따로 해야 하는 불편함이 발생한다.
rest.get("https://dev.melon.com/api/album/song/list", (req, res, ctx) => {
  return res(ctx.status(200), ctx.json({}));
});

rest.get("https://sandbox.melon.com/api/album/song/list", (req, res, ctx) => {
  return res(ctx.status(200), ctx.json({}));
});
이 불편함을 어떻게 해소할 수 있는지 살펴보자.

Tagged Template Literal을 활용한 해결 방법
필자는 간단히 Tagged Template Literal을 활용하여 Node.js에서 구동될 때만 prefix로 호스트를 붙여 주었다. 그러면 애플리케이션 코드와 테스트 코드에서는 여전히 상대 경로로 url을 작성할 수 있게 된다. 코드를 보면 Node.js인 경우만 prefix를 붙이는 것을 볼 수 있다.

import isNode from "detect-node";

const host = isNode ? "http://localhost" : "";

export function testUrl(strings, ...values) {
  // from https://github.com/mkrause/uri-tag/blob/25311b195fcffd2bae943fe677622538794be9f7/src/uri.js#L3
  const relativeUrl = values.reduce((prev, cur, i) => `${prev}${cur}${strings[i + 1]}`, strings[0]);

  return `${host}${relativeUrl}`;
}
이 함수를 활용하여 Template Literal을 사용하면 여전히 상대경로를 사용해서 코드를 깔끔하게 유지할 수 있다.

import { testUrl } from "shared/util/tags";

// 애플리케이션 코드에서 사용
const query = useQuery([testUrl`/api/album/song/list`, albumId], () => fetchSampleAlbum(albumId));

// 테스트 코드에서 사용
rest.get(testUrl`/api/album/song/list`, (req, res, ctx) => {
  return res(ctx.status(200), ctx.json({}));
});

// 브라우저에 처리 - '/api/album/song/list'
// Node.js에서 처리 - 'http://localhost/api/album/song/list'
마치며
이 방식은 MSW 모킹이 적용된 스토리 컴포넌트를 Jest 테스트를 활용한 자동화 테스트에 적용하기 위한 목적으로 연구되었다. 스토리에 필요한 모킹이 이미 처리되어 있으므로 자동화 테스트 시 또 한번 중복 모킹을 해야 하는 불편함을 해소하고 싶었다. msw-storybook-addon, @storybook/testing-react 도구만으로 다양한 모킹을 충분히 재사용할 수 있으나 MSW 모킹이 중복으로 일어나는 부분은 Custom Render를 활용하여 테스트 코드를 깔끔하게 정리하였다.

깔끔한 코드, 잘 읽히는 코드, 중복을 최소한 코드, 관심사의 분리 등 개발자들의 깔끔함(?)을 위한 노력은 오늘도 계속된다. 소소하지만 꿀팁이 되었으면 좋겠다.
