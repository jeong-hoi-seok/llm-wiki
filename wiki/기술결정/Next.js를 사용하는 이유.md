# Next.js를 사용하는 이유

## 결정
웹 프로젝트에서 [[React]]만으로 앱을 구성하기보다, 라우팅·렌더링·데이터 패칭·빌드 최적화가 함께 필요한 경우 [[Next.js]]를 기본 후보로 둔다.

이 문서는 모든 웹 프로젝트에 Next.js를 강제하는 문서가 아니다. 실제 채택 여부는 프로젝트 `docs/`의 아키텍처 결정에서 확정한다.

## 배경
React는 UI 라이브러리다. 실제 제품을 만들 때는 라우팅, 서버 렌더링, 데이터 패칭, 번들링, 이미지/폰트 최적화, API 엔드포인트 같은 주변 결정이 필요하다.

Next.js는 이 주변 결정을 프레임워크 단에서 제공한다. 팀은 낮은 수준의 도구 조합보다 제품 기능과 페이지 설계에 더 집중할 수 있다.

## 선택지

| 선택지 | 장점 | 단점 |
|---|---|---|
| React + Vite | 가볍고 자유도 높음 | 라우팅·SSR·데이터 전략을 직접 조합해야 함 |
| Next.js | 라우팅·렌더링·최적화 통합 | 프레임워크 규칙과 서버/클라이언트 경계를 학습해야 함 |
| Remix / React Router 기반 | 웹 표준 폼·라우팅에 강함 | 팀 경험과 생태계 기준 검토 필요 |

## 판단 기준
- 페이지 기반 웹 앱인가?
- SEO, 공유 링크, 초기 로딩 성능이 중요한가?
- 서버에서 먼저 렌더링해야 하는 화면이 있는가?
- 파일 기반 라우팅과 중첩 레이아웃이 생산성을 높이는가?
- 서버 컴포넌트와 클라이언트 컴포넌트 경계를 팀이 운영할 수 있는가?
- 배포 환경이 Next.js 런타임과 잘 맞는가?

## 최종 선택
다음 조건이면 Next.js를 선택한다.

- 라우트가 명확한 웹 서비스
- SEO나 초기 렌더링 품질이 중요한 서비스
- 대시보드, 관리자, 마케팅, 문서, 커머스처럼 페이지 구조가 커질 서비스
- 서버 렌더링, 정적 생성, 스트리밍, API route 같은 기능을 한 프레임워크 안에서 관리하고 싶은 서비스

다음 조건이면 React + Vite 같은 가벼운 구성을 먼저 고려한다.

- 순수 클라이언트 앱
- SEO가 중요하지 않은 내부 도구
- 정적 SPA로 충분한 작은 프로젝트
- 서버/클라이언트 경계가 오히려 복잡도를 높이는 프로젝트

## 선택 이유

### 1. React 제품 개발에 필요한 주변 기능을 통합한다
Next.js는 React 위에 라우팅, 렌더링, 데이터 패칭, 번들링, 최적화를 얹은 프레임워크다. 별도 도구 조합을 매번 설계하는 비용을 줄인다.

### 2. App Router가 파일 구조와 URL 구조를 연결한다
`app/` 폴더 구조가 URL 구조가 된다. `layout`, `page`, `loading`, `error`, `route` 같은 특수 파일로 화면 구조와 상태를 명시할 수 있다.

### 3. 서버 컴포넌트 기본값이 클라이언트 번들을 줄인다
상호작용이 없는 UI와 데이터 패칭은 서버 컴포넌트에 두고, 상태·이벤트·브라우저 API가 필요한 부분만 클라이언트 컴포넌트로 둔다. 이 경계는 성능과 책임 분리에 도움이 된다.

### 4. 렌더링 전략을 페이지 특성에 맞게 선택할 수 있다
정적 생성, 서버 렌더링, 재검증, 스트리밍을 한 앱 안에서 섞어 쓸 수 있다. 화면마다 데이터 변경 주기와 사용자 맥락이 다를 때 유리하다.

### 5. 생태계와 배포 경험이 풍부하다
Next.js는 React 기반 웹 앱에서 널리 쓰이며, [[Tailwind CSS]], [[shadcn-ui]], [[FSD 폴더구조]] 같은 구조와 조합하기 쉽다.

## 포기한 것
- React + Vite 대비 설정 자유도가 줄어든다.
- 서버/클라이언트 컴포넌트 경계를 잘못 잡으면 복잡해진다.
- Next.js 규칙과 배포 런타임에 영향을 받는다.
- 단순 SPA에는 과할 수 있다.

## 에이전트 적용 기준
- 프로젝트 `docs/`에 Next.js 채택 결정이 있으면 그 규칙을 우선한다.
- 채택 결정이 없으면 `Next.js가 필요한 이유`를 SEO, 라우팅, 렌더링, 데이터 흐름 기준으로 먼저 검토한다.
- Next.js를 쓰더라도 모든 컴포넌트를 클라이언트 컴포넌트로 만들지 않는다.
- `src/app` 라우트와 나머지 앱 구조는 [[FSD 폴더구조]] 같은 아키텍처 문서와 함께 판단한다.
