# 응집도

## 한 줄 요약
**수정되어야 할 코드가 항상 같이 수정되는지**. 응집도가 높으면 한 부분 수정 시 다른 부분에서 의도치 않은 오류가 안 남 — 함께 바뀔 것이 구조적으로 묶여 있기 때문. [[코드 품질]] 4기준 중 하나.

## 전략
- **함께 수정되는 파일을 같은 디렉토리에** — 흩어두면 하나 고치고 다른 걸 빠뜨림. (raw: `examples/code-directory.md`) → [[FSD 폴더구조|FSD]]의 slice 응집과 동일 원리.
- **매직 넘버 없애기(응집 관점)** — 같은 의미의 상수를 한 곳에서 관리. (raw: `examples/magic-number-cohesion.md`)
- **폼의 응집도 생각하기** — 필드 단위 vs 폼 단위 응집 중 선택. (raw: `examples/form-fields.md`)

## 빠른 참조

| 코드 냄새 | 개선 방법 |
|---|---|
| 같은 매직 넘버 3곳 이상 | 공유 상수로 추출 |
| 기능이 `components/`, `hooks/`, `types/` 모두 수정 | `domains/featureName/`으로 배치 |
| 긴 상대 경로 `../../..` | 파일들을 가까이 배치 |
| 타입 정의가 사용처와 멀리 떨어짐 | 사용하는 코드와 함께 배치 |

## 코드 예제

### 함께 수정되는 파일을 같은 디렉토리에

❌ 종류별 분리 (의존 관계 파악 어려움):
```
src/
├── components/UserForm.tsx
├── hooks/useUserValidation.ts
├── types/userTypes.ts
└── api/userApi.ts
```

✅ 도메인별 배치:
```
src/
├── components/   # 전역 공유
└── domains/User/
    ├── UserForm.tsx
    ├── useUserValidation.ts
    ├── types.ts
    └── api.ts
```

### 매직 넘버 제거

❌ 같은 숫자가 분산:
```tsx
// Pagination.tsx
const pages = Math.ceil(total / 20);
// useItems.ts
const offset = (page - 1) * 20;
```

✅ 상수로 통합:
```tsx
export const PAGE_SIZE = 20;
import { PAGE_SIZE } from './constants';
```

## 주의 (트레이드오프)
- 응집을 위한 공통화·추상화는 [[가독성|가독성]]을 떨어뜨릴 수 있음.
  - 함께 안 고치면 버그 → **응집도 우선**.
  - 위험 낮음 → 가독성 우선(중복 허용, [[결합도|결합도]]↓).
- 중복이 나은 경우: 두 기능이 다르게 발전할 가능성이 높을 때 / 공유 코드가 5줄 미만일 때.
