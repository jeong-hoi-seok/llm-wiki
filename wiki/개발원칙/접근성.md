# 접근성

## 한 줄 요약
모든 사용자(장애인·키보드 사용자·일반)가 웹을 쓸 수 있게 하는 기본 원칙. 프론트엔드가 HTML 구조·인터랙션을 만들므로 **접근성의 출발점**. (토스 A11y Fundamentals) 상위: [[index]]

## 왜 중요한가
- **장애인**: 스크린 리더가 `role`·`label`·`alt` 없으면 요소를 못 읽음. `<div onclick>`은 버튼으로 인식 안 됨.
- **일반 사용자**: 시맨틱 마크업이 익숙한 기본 동작 제공 — `<a>` 새 창/링크 복사, `<form>` Enter 제출, 키보드(Tab) 탐색.
- **개발자**: `getByRole`로 테스트 요소 특정이 쉬워짐 (querySelector·`data-testid`보다 견고).

## 스크린 리더 3요소
화면 요소를 이 순서로 읽음:
- **역할 (Role)** — 요소 종류 (버튼·입력창·스위치)
- **레이블 (Label)** — 이름 (무슨 기능인지)
- **상태 (State)** — 현재 상태 (활성·꺼짐·선택됨)

## 4대 원칙

### 1. 🧱 올바른 구조
- **버튼 안에 버튼/링크 넣지 않기** — HTML은 인터랙티브 요소 중첩 불허. `<a><Button/></a>` ❌ → `<Button as="a" href>`. (raw: `structure/button-inside-button.md`)
- 테이블 행 전체 클릭. (raw: `structure/table-row-link.md`)

### 2. 🗣️ 의미 정확히 전달
- **인터랙티브 요소에 이름 붙이기** — input·button·select 등은 목적을 알리는 이름 필수. (raw: `semantic/required-label.md`)
- 같은 이름 요소엔 설명 추가. (raw: `semantic/duplicate-interactive-element.md`)

### 3. 🎯 예측 가능한 인터랙션
- **가짜 버튼 금지** — `<div cursor:pointer onclick>`은 키보드·스크린리더에서 버튼이 아님. 진짜 `<button>` 사용. (raw: `predictability/fake-button.md`) → [[예측 가능성]]
- 입력 요소는 `<form>`으로 감싸기 (Enter 제출). (raw: `predictability/form.md`)

### 4. 🌈 시각 정보에만 의존하지 않기
- 색·이미지·아이콘은 텍스트/대체 수단으로 보완. 이미지·아이콘에 `alt`. (raw: `alt-text/image-alt.md`)

## 실무 적용 기준
- 클릭 가능한 건 `<button>`/`<a>`로 (div+onclick ❌).
- 폼은 `<form>` + `<input>` + `<button type="submit">`.
- **eslint**로 코드 단계에서 접근성 위반 사전 탐지 (raw: `eslint/rules.md`).
- 테스트는 `getByRole` 기준.
