# 프로젝트 맥락 위키 작성 가이드

개별 프로젝트 안에 **맥락 파악용** 위키를 만드는 방법. 상위: [[index]]

> **이 문서의 용도** — 이 위키(Obsidian) 내부 운영 규칙이 아니다. 사용자가 **다른 프로젝트에서 에이전트에게 작업 내용을 학습·문서화**시킬 때 따르는 규칙이다.
>
> **전체 위키가 아니라 맥락 위키다.** API 명세·코드 구조·튜토리얼 같은 전체 문서화가 아니라, **도메인과 그동안 내가(팀이) 한 일·결정**을 위주로 서술한다. 목적은 새 세션의 에이전트가 와도 프로젝트의 도메인 맥락과 결정 이력을 빠르게 이해하는 것이다. 코드로 알 수 있는 사실은 적지 않고, 코드만 봐서는 모르는 맥락만 남긴다.

## 실행 원칙 — 사용자 명령이 있을 때만

**에이전트가 먼저 제안하거나 자동으로 작성하지 않는다.** 사용자가 "이렇게 기록해라 / 문서화해라"라고 명령하기 전까지는 실행하지 않는 것이 원칙이다.

- 작업을 했다고 해서 `decisions/`·`worklog.md`를 임의로 만들거나 갱신하지 않는다.
- `wiki/` 폴더 생성, 파일 작성, 압축 모두 사용자가 원할 때만 한다.
- 의미 있는 결정이 있었다고 판단되면 먼저 작성을 제안하는 것이 아니라, 기록할지는 사용자 판단에 맡긴다. (이 위키의 다른 운영 문서와 달리 자동 갱신 대상이 아니다.)

## 폴더 구조

프로젝트 root에 `wiki/`가 없으면 만든다. 원문까지 보존해야 하면 프로젝트 root에 `raw/`도 둔다.

`raw/`와 `wiki/`의 역할은 분리한다.

- `raw/` — 회의록 원문, 작업 메모 원문, 긴 대화 로그처럼 다시 해석할 수 있어야 하는 원본. 수정하지 않는다.
- `wiki/summaries/` — raw를 읽기 좋게 압축한 요약본. 에이전트가 기본으로 먼저 읽는 자료.
- `wiki/decisions/` — 현재 프로젝트 판단에 영향을 주는 결정·오버라이드 기록.
- `wiki/worklog.md` — 무엇을 했는지 빠르게 훑는 누적 로그.

구조는 다음과 같다.

```
raw/
├── meetings/         # 회의록 원문
│   ├── 2026-06-24-결제-정책-회의.md
│   └── ...
└── work/             # 작업 메모·긴 작업 로그 원문
    ├── 2026-06-24-결제-모듈-작업.md
    └── ...

wiki/
├── summaries/        # raw 요약본
│   ├── 2026-06-24-결제-정책-회의.md
│   └── ...
├── decisions/        # 날짜별 결정·맥락 이력
│   ├── 2026-06-23.md
│   └── ...
└── worklog.md        # 누적 작업 로그 (온디맨드)
```

- `wiki/`가 이미 있으면 기존 구조를 깨지 않고 필요한 폴더만 보강한다.
- raw가 없거나 원문 보존 가치가 낮으면 `raw/`를 억지로 만들지 않는다.

## raw/ — 원문 보관소

회의록 원문, 작업 메모 원문, 긴 대화 로그처럼 나중에 다시 검증할 수 있어야 하는 자료를 보관한다.

- raw는 수정하지 않는다. 오탈자나 표현이 거칠어도 원문성을 유지한다.
- 파일명은 날짜 + 주제로 쓴다: `YYYY-MM-DD-주제.md`.
- 회의록 원문은 `raw/meetings/`에 둔다.
- 작업 메모·긴 작업 로그 원문은 `raw/work/`에 둔다.
- 보안 정보, 토큰, 개인정보는 raw에 넣지 않는다. 필요하면 마스킹한다.

raw는 기본 학습 대상이 아니다. 에이전트는 요약본과 최신 결정 문서를 먼저 읽고, 아래 상황에서만 raw를 확인한다.

- 요약본만으로 판단 근거가 부족할 때
- 특정 발언, 수치, 합의 여부를 검증해야 할 때
- 요약본끼리 충돌하거나 최신 결정과 어긋날 때
- 사용자가 원문 확인을 명시적으로 요청했을 때

## summaries/ — raw 요약본

raw 원문을 에이전트가 빠르게 읽을 수 있게 압축한 문서다. 회의록 원문을 요약한 md는 `wiki/summaries/`에 둔다.

요약본은 원문을 대체하는 기본 진입점이다. 다만 최종 진실은 아니다. 이후 결정이 바뀌면 `decisions/`의 최신 오버라이드가 우선한다.

권장 구조:

```md
# 2026-06-24 결제 정책 회의

## TL;DR
- 회의 결론 1~3줄.

## 결정
- 결정된 내용. 없으면 `없음`.

## 근거
- 왜 그렇게 판단했는지.

## 열린 질문
- 아직 확정되지 않은 것.

## 액션 아이템
- 담당자 / 할 일 / 기한.

## 원문
- `raw/meetings/2026-06-24-결제-정책-회의.md`
```

작성 기준:

- 원문을 전부 재작성하지 않는다. 에이전트가 다음 작업에 필요한 맥락만 남긴다.
- 결정·근거·열린 질문·액션 아이템을 분리한다.
- 확정되지 않은 내용은 결정처럼 쓰지 않는다.
- 추론은 `[추론]`으로 표시한다.
- 원문 경로를 반드시 남긴다.

## worklog.md — 누적 작업 로그

무엇을 했는지 **아주 간단히** 누적 기록하는 로그다.

- **온디맨드** — 매 작업마다 자동으로 쓰지 않는다. 사용자가 기록을 요청할 때 추가한다.
- 한 항목은 한 줄. 날짜 + 핵심만. 구현 세부는 적지 않는다.
- 최신 항목이 위로 오게 쌓는다.

```md
# Worklog

- 2026-06-23 — 결제 모듈 토스페이먼츠로 교체
- 2026-06-22 — 로그인 토큰 만료 버그 수정
```

- **압축** — 길어지면 콤팩트하게 줄인다. 오래된 항목을 주·월 단위로 묶어 한 줄 요약으로 합치고, 세부가 필요하면 해당 `decisions/` 날짜를 가리킨다.

## decisions/ — 날짜별 결정·맥락 이력

그 날 무엇을 했고 **왜 그렇게 결정했는지**, 프로젝트 맥락을 정리하는 곳이다. 파일명은 `YYYY-MM-DD.md`.

> **읽기 원칙 — 최신순 우선.** 맥락 파악은 **반드시 최신 날짜 파일부터** 본다. 최우선순위가 최신순이다. 오버라이드가 과거 결정을 덮으므로 최신 파일이 현재 살아있는 상태를 가장 정확히 반영한다. 오래된 파일은 "왜 그렇게 됐는지" 배경이 필요할 때만 거슬러 올라간다.

- 그 날의 결정·배경·트레이드오프·바뀐 맥락을 적는다. worklog가 "무엇"이라면 decisions는 "왜"다.
- **과거 날짜 파일은 절대 수정하지 않는다.** 이미 일어난 결정의 기록이기 때문이다.
- 결정이 바뀌면 과거 파일을 고치지 말고, **현재 날짜 파일에 오버라이드로 기록한다.** "이전에는 A였는데, 지금은 B다. 이유는 …" 형태로 변경 전·후와 이유를 남긴다.

```md
# 2026-06-23

## 결정 | 상태관리 라이브러리 변경
- 기존: Redux Toolkit
- 변경: Zustand
- 이유: 보일러플레이트 과다, 서버 상태는 React Query가 이미 담당 → 클라이언트 상태만 가볍게.
- 영향: store 구조 재작성, 기존 selector 패턴 폐기.

## 오버라이드 | 2026-06-10 결정 보정
- 2026-06-10에 "API 캐시는 SWR" 로 정했으나, React Query로 변경.
- 이유: mutation·낙관적 업데이트 사용처가 늘어 React Query 생태계가 더 맞음.
```

### 압축 (파일이 많아질 때)

날짜 파일이 쌓이면 **오래된 연속 구간을 하나로 묶는다.** 파일명은 범위로 적는다: `YYYY-MM-DD ~ YYYY-MM-DD.md`.

- 예: `2026-06-01.md` ~ `2026-06-14.md` 14개 → `2026-06-01 ~ 2026-06-14.md` 1개.
- **결정과 오버라이드의 핵심은 보존한다.** 압축은 사소한 서술·중복을 줄이는 것이지 결정 이력을 지우는 게 아니다. 최종적으로 살아있는 결정만 남기되, 무엇이 언제 뒤집혔는지는 유지한다.
- 묶인 파일 안에서는 날짜 소제목(`## 2026-06-03`)으로 시간순을 보존한다.
- 최근 구간은 묶지 않고 날짜별로 둔다. 활성 맥락은 세밀하게, 오래된 맥락은 거칠게.

## 왜 이렇게 나누나

- **decisions append-only** — 과거 결정을 덮어쓰면 "왜 그때 그렇게 했는지"가 사라진다. 새로 온 에이전트·사람이 맥락을 추적하려면 결정의 시간순 이력이 보존돼야 한다. 그래서 수정 대신 현재 날짜 오버라이드를 쓴다.
- **worklog 분리** — 결정 이력에 사소한 작업까지 섞으면 맥락이 묻힌다. 가벼운 작업 흐름은 worklog가, 의미 있는 판단은 decisions가 맡는다.

## 에이전트 적용 순서

**읽기**는 맥락 파악을 위해 자유롭게, **쓰기**는 사용자 명령이 있을 때만.

1. 작업 맥락 파악이 필요하면 `decisions/`를 먼저 **최신순으로** 읽는다. 최신 결정과 오버라이드가 현재 기준이다.
2. 이어서 `worklog.md`로 최근 작업 흐름을 훑고, 필요한 경우 `summaries/`를 최신순으로 읽는다.
3. raw 원문은 기본으로 읽지 않는다. 요약본만으로 부족하거나 근거 검증이 필요할 때만 `raw/`를 확인한다.
4. 사용자가 기록·문서화를 명령하면, 프로젝트에 `wiki/`가 있는지 확인하고 없으면 만든다. 원문 보존이 필요할 때만 `raw/`도 만든다.
5. 회의록·작업 메모 원문 정리 명령이면 원문은 `raw/`에 두고, 요약본은 `wiki/summaries/`에 만든다.
6. 결정 기록 명령이면 오늘 날짜 `decisions/YYYY-MM-DD.md`에 기록한다. 과거 파일은 수정하지 않는다.
7. 작업 로그 명령이면 `worklog.md`에 한 줄로 추가한다.
8. 명령이 없으면 아무것도 만들거나 고치지 않는다.
