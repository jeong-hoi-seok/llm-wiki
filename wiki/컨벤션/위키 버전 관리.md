# 위키 버전 관리

## 한 줄 요약

이 위키의 버전은 `wiki/index.md` frontmatter의 `version` 값으로 관리한다. 현재 버전은 `0.5.2`이다.

## 왜 `0.3.0`으로 시작하나

`0.1.0`은 초기 스캐폴드 수준에 가깝다. 현재 위키는 이미 `raw/`, `wiki/`, `context/`, `log/`, `AGENTS.md` 운영 규칙과 주요 프론트엔드 문서를 갖췄다.

다만 아직 1.0은 아니다.

- 팀 공유용 경로 resolver는 문서화됐지만 실제 팀 적용 검증이 없다.
- 끊어진 링크·고립 문서·출처 검증 자동화가 없다.
- 기술 범위가 Next.js·Expo·FSD·테스트 일부에 치우쳐 있다.
- 버전 정책도 이제 막 생겼다.

그래서 현재 상태는 **사용 가능한 초안 기준선**으로 보고 `0.3.0`에서 시작한다.

## 버전 형식

형식은 SemVer를 빌려온다.

```txt
MAJOR.MINOR.PATCH
```

1.0 전까지는 `0.MINOR.PATCH`를 사용한다. `1.0.0`은 팀 공유 기준선이 충분히 안정됐을 때만 올린다.

## 올리는 기준

여러 변경이 한 작업에 섞이면 가장 큰 기준 하나만 적용한다.

`minor`는 **드물게** 올린다. 1.0 전까지 `minor`는 곧 마일스톤이므로 가볍게 올리지 않는다. 하루에 `minor`가 2회 이상 발생하면 기준을 잘못 적용했을 가능성이 높으니 다시 검토한다.

### `0.x.0`의 `x` (MINOR) — 아래 셋 중 하나를 충족할 때만

`x`는 **위키가 다루는 범위·능력의 의미 있는 확장**, 또는 **에이전트 작동 방식의 구조 변경**을 뜻한다. "전엔 못 하던 걸 하게 됨" 수준이다.

1. **새 카테고리 신설 + 실제 문서 3개 이상.** 카테고리만 만들고 1~2개면 `patch`. 빈 카테고리는 `minor`가 아니다.
2. **대형 ingest.** 새 raw 묶음을 처리해 개념 문서를 3개 이상 신설하고 index에 연결.
3. **작동 방식 구조 변경.** 판정 질문 = "이 변경 후 에이전트가 실제로 다르게 동작하나?" YES면 `minor`.
   - 예: ingest 흐름 단계 추가·삭제, 폴더 역할 재정의, 경로 resolver 도입, 새 필수 선행 읽기 규칙.

### PATCH — 그 외 파일 변경 전부

| 변경 | 예시 | 버전 |
|---|---|---|
| 기존 컨벤션 보강 | 절차·하위 항목·예외 추가 (예: 동기화 지침) | patch |
| 문서 이동·분류·이름 변경 | 폴더명·파일명 변경, `원문요약/`→`summaries/` | patch |
| 단일 문서 신설 | 새 문서 1~2개, 워크플로 변화 없음 | patch |
| 기존 판단 기준 조정 | 기술결정 결론 변경, 기존 컨벤션 문구 수정 (스코프 안) | patch |
| 기존 문서 보강 | 설명·예시·출처 추가 (양이 많아도 patch) | patch |
| 오탈자·표현 수정 | 의미 변화 없는 문장 정리 | patch |
| 링크·index·log 보정 | 누락 링크 추가, 오래된 목차 갱신 | patch |
| 질문 답변 | 파일 변경 없는 답변 | 변경 없음 |

### 경계 케이스

| 상황 | 판정 | 이유 |
|---|---|---|
| 새 컨벤션 문서 1개 추가, 워크플로 안 바뀜 | patch | 참고용 추가일 뿐 |
| 새 컨벤션 + AGENTS 흐름에 강제 단계 생김 | minor | 작동 방식 변경(#3) |
| 카테고리 만들고 문서 2개 | patch | 3개 미만 |
| 기존 문서 10개 대량 보강 | patch | 스코프 확장 아님 (양 ≠ minor) |
| 기술결정 결론을 뒤집음 | patch | 단일 판단, 범위 그대로 |

### 배칭·릴리즈 케이던스

- `patch`는 **누적**한다. 매 편집마다 tag·release를 만들지 않는다.
- 릴리즈 시점: `minor`가 발생할 때, 또는 `patch`가 여러 개 쌓여 한 묶음으로 의미가 생길 때.
- 변경이 섞이면 가장 높은 기준 하나만 적용한다.

## 원격 버전 동기화

버전 작업 전에는 GitHub Release 최신 tag와 `wiki/index.md`의 `version`이 같은지 확인한다.

- GitHub Release tag는 `vX.Y.Z` 형식이다.
- `wiki/index.md`의 `version`은 `X.Y.Z` 형식이다.
- 비교할 때는 `v${version}`으로 맞춰 본다.

둘이 맞지 않으면 로컬 기준이 오래됐을 가능성이 있으므로 먼저 원격 정보를 최신화한다.

```sh
git fetch origin main --tags
```

원격 `origin/main`의 `wiki/index.md`가 더 최신이면 로컬 작업 전 아래 명령으로 갱신한다.

```sh
git pull --ff-only origin main
```

이미 로컬 변경이 있거나 fast-forward가 안 되면 임의로 reset/rebase하지 않는다. 현재 `index.version`, GitHub Release tag, `origin/main`의 `index.version`을 확인해 사용자에게 동기화 필요 상태를 먼저 보고한다.

## 1.0.0 기준

아래 조건을 만족하면 `1.0.0` 검토 가능.

- `AGENTS.md`와 주요 프로젝트용 `AGENTS.md 작성 가이드`가 팀 공유 방식까지 포함한다.
- `wiki/index.md`가 핵심 문서를 빠짐없이 가리킨다.
- 주요 문서가 출처, 플랫폼, 상태를 가진다.
- 링크 lint 기준이 정해져 있고 큰 깨짐이 없다.
- 공통 지식과 프로젝트 맥락 문서 역할이 명확히 분리돼 있다.

## 작업 절차

1. 변경 범위를 확인한다.
2. GitHub Release 최신 tag와 `wiki/index.md`의 `version`을 비교한다.
3. 둘이 맞지 않으면 `git fetch origin main --tags`로 최신화하고, 필요 시 `git pull --ff-only origin main`을 실행한 뒤 다시 확인한다.
4. 이 문서의 기준으로 `minor` 또는 `patch`를 고른다.
5. `wiki/index.md` frontmatter의 `version`과 `updated`를 갱신한다.
6. `wiki/log/YYYY-MM-DD.md`에 이전 버전, 새 버전, bump 이유를 남긴다.
7. `wiki/log.md` 날짜별 요약도 갱신한다.
8. 변경 커밋을 만든다.
9. `vX.Y.Z` 형식으로 git tag를 만든다. 예: `v0.5.0`
10. branch와 tag를 push한다.
11. GitHub Release를 만든다. 릴리즈 제목은 tag와 같게 쓰고, 본문에는 `wiki/log/YYYY-MM-DD.md`의 버전 변경 요약을 압축해 적는다.
