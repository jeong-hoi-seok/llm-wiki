# Component-Driven Development

## 한 줄 요약
Component-Driven Development는 화면을 한 번에 만들기보다 작은 컴포넌트와 상태를 독립적으로 만들고 조합해 제품 화면으로 확장하는 개발 방식이다.

## 핵심 내용
CDD에서는 컴포넌트가 독립적으로 렌더링되고 검증될 수 있어야 한다. Storybook은 이 방식의 대표 도구다.

29CM 글에서 CDD는 Storybook을 쓰면서 자연스럽게 따라오는 개발 방식으로 설명된다. 전체 페이지를 먼저 띄워 맞추기보다 가장 기본적이고 독립적인 컴포넌트부터 개발한다. 이 과정은 props 설계가 특정 페이지 맥락에 과하게 묶이는 것을 줄인다.

## 실무 적용 기준
- 작은 UI 단위부터 Story를 만든다.
- 각 컴포넌트의 주요 상태를 Story로 남긴다.
- API가 필요한 컴포넌트는 MSW로 상태를 표현한다.
- 디자인 시스템 컴포넌트는 시각적 회귀 테스트와 연결한다.
- 페이지 컴포넌트도 주요 사용자 상태를 Story로 관리한다.
- 컴포넌트가 독립 실행되지 않으면 의존성이 과한지 점검한다.
- Story 작성이 어렵다면 props API나 데이터 의존 방식이 복잡한 신호일 수 있다.

## 장점
- 중복 컴포넌트 생성을 줄인다.
- 리뷰어가 UI 동작을 쉽게 확인한다.
- QA와 개발이 같은 상태를 기준으로 대화한다.
- 테스트 시나리오가 자산화된다.
- 신규 입사자가 컴포넌트 목록과 사용법을 빠르게 파악한다.
- 디자이너가 개발자 로컬 환경 없이 구현 상태를 확인한다.
