# 사용자 시나리오 기반 테스트

## 한 줄 요약
사용자 시나리오 기반 테스트는 사용자가 보는 화면, 입력하는 행동, 이후 바뀌는 결과를 하나의 흐름으로 검증한다.

## 핵심 내용
프론트엔드 테스트는 내부 함수 호출보다 사용자 경험의 흐름을 잘 설명해야 한다.

예시 흐름:
- 사용자가 약관을 모두 체크한다.
- 주문 버튼이 활성화된다.
- 사용자가 쿠폰을 선택한다.
- 결제 금액이 바뀐다.
- 오류 응답을 받으면 오류 메시지가 보인다.

우아한형제들 단위 테스트 글의 Given-When-Then도 같은 방향이다. Given은 화면과 조건을 준비하는 단계, When은 사용자 상호작용, Then은 사용자가 볼 결과다. 통합 테스트에서는 여기에 API 응답과 여러 컴포넌트 연동이 추가된다.

## 실무 적용 기준
- 테스트명은 사용자 행동과 기대 결과를 함께 담는다.
- `userEvent`로 클릭, 입력, 선택을 수행한다.
- `screen.getByRole`, `findByText` 등 화면 기준 query를 우선한다.
- 한 테스트는 하나의 의미 있는 사용자 흐름만 검증한다.
- 시나리오 이름에는 사용자 행동과 결과를 같이 적는다.
- 테스트가 너무 길어지면 정책 검증은 단위 테스트로 내리고 흐름 검증만 남긴다.

## 나쁜 신호
- 테스트명이 “컴포넌트가 정상 작동한다”다.
- 사용자 행동 없이 상태를 직접 바꾼다.
- 여러 사용자 시나리오가 한 테스트에 섞여 실패 원인을 알기 어렵다.
- API 성공, 빈 응답, 오류 응답이 한 케이스에 섞여 있다.
