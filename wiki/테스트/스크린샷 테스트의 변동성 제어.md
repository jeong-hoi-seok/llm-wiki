# 스크린샷 테스트의 변동성 제어

## 한 줄 요약
스크린샷 테스트는 렌더링 시점, 시간, 애니메이션, 폰트, 네트워크, 실행 환경을 통제해야 신뢰할 수 있다.

## 핵심 내용
스크린샷 테스트는 작은 차이에도 실패한다. 이 민감함은 장점이지만 환경이 흔들리면 flaky test가 된다.

우아한형제들 원문은 두 가지 대표 문제를 다룬다.

- 화면 렌더링이 끝나기 전에 찍혀 빈 화면이 스크린샷으로 남는 문제
- 자동 슬라이드처럼 `setTimeout` 기반 UI가 실행 시점마다 다른 화면을 만드는 문제

CSR 환경에서는 `load`, `domContentLoaded`만으로 React 렌더링 완료를 보장하기 어렵다. 원문은 MutationObserver로 DOM 변화가 멈출 때까지 기다리는 방식을 사용했다.

## 통제 대상
- 렌더링 완료 전 촬영
- 타이머와 자동 슬라이드
- 애니메이션과 transition
- 외부 폰트 로딩
- 랜덤 값과 현재 시간
- OS/브라우저/font rendering 차이
- 네트워크 지연과 이미지 로딩

## 실무 적용 기준
- CI의 고정 Docker image에서 실행한다.
- 애니메이션은 테스트 환경에서 끈다.
- 현재 시간과 랜덤 값을 고정한다.
- 렌더링 완료 신호를 기다린 뒤 촬영한다.
- 기준 이미지 갱신은 자동이 아니라 리뷰 가능한 수동 흐름으로 둔다.

## CI 기준
- 로컬보다 CI에서 실행하는 편이 환경 차이를 줄인다.
- Playwright Docker image처럼 실행 환경을 고정한다.
- 리포트는 MR/PR에서 바로 열 수 있게 한다.
- snapshot update는 수동 job으로 분리한다.
