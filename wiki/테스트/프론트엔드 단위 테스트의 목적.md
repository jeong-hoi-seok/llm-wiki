# 프론트엔드 단위 테스트의 목적

## 한 줄 요약
프론트엔드 단위 테스트의 목적은 함수, 훅, 컴포넌트의 작은 단위가 **외부에서 기대한 동작**을 안정적으로 수행하는지 빠르게 검증하는 것이다.

## 핵심 내용
단위 테스트는 내부 구현을 증명하는 코드가 아니다. 사용자가 접근 가능한 결과, 함수의 공개 계약, 컴포넌트의 렌더링 결과와 상호작용 결과를 검증해야 한다.

좋은 단위 테스트는 다음 역할을 한다.

- 변경 후 기존 동작이 깨졌는지 빠르게 알려준다.
- 코드가 어떤 정책을 만족해야 하는지 명세로 남긴다.
- 리팩터링할 때 안전망이 된다.
- 작은 단위의 실패 원인을 빠르게 좁힌다.

## FIRST 원칙
우아한형제들 원문은 단위 테스트의 기본 특성으로 F.I.R.S.T를 소개한다.

- Fast: 빠르게 실행된다.
- Isolated: 외부 요인에 종속되지 않는다.
- Repeatable: 반복 실행해도 같은 결과를 낸다.
- Self-validating: 테스트 자체가 성공/실패를 판단한다.
- Timely/Thorough: 적절한 시점에 작성하거나, 정상 흐름과 비정상 흐름을 충분히 다룬다.

## 실무 적용 기준
- 순수 함수는 입력과 출력 중심으로 테스트한다.
- 훅은 반환값과 상태 전이처럼 공개된 사용 방식 중심으로 테스트한다.
- 컴포넌트는 사용자가 보는 텍스트, role, 상태, 이벤트 결과 중심으로 테스트한다.
- 라이브러리 자체 동작을 다시 검증하지 않는다.
- 테스트명은 “정상 동작한다”가 아니라 “어떤 조건에서 어떤 결과가 나와야 하는지”를 적는다.
- 단위 테스트가 복잡해지면 해당 단위가 너무 많은 책임을 가진 것은 아닌지 본다.
