# 내부 구현을 테스트하면 안 되는 이유

## 한 줄 요약
내부 구현을 테스트하면 사용자 동작은 그대로인데 테스트만 깨지는 일이 늘고, 리팩터링을 방해하는 테스트가 된다.

## 핵심 내용
프론트엔드 테스트는 컴포넌트 내부 store, hook 호출 순서, private 함수 구조보다 외부 동작을 우선 검증해야 한다.

예를 들어 버튼 클릭 후 Zustand store 값이 바뀌는지 직접 확인하는 테스트는 컴포넌트가 아니라 상태 관리 라이브러리와 내부 구현을 검증한다. 더 좋은 테스트는 사용자가 버튼을 클릭했을 때 heading 문구가 의도한 값으로 바뀌는지 확인하는 방식이다.

원문 사례의 핵심은 “바나나라는 단어가 화면 어딘가에 있다”도 충분히 좋은 검증이 아니라는 점이다. 테스트 목적이 heading 문구 변경이라면 `heading` role과 정확한 accessible name으로 검증해야 한다. 검증 범위가 흐리면 우연히 통과하는 테스트가 된다.

## 실무 적용 기준
- store 상태 직접 검증보다 화면 변화 검증을 우선한다.
- `data-testid`는 접근 가능한 role/name으로 찾기 어려운 경우에만 사용한다.
- 구현 함수 호출 여부보다 사용자 결과를 우선한다.
- 리팩터링 후 사용자 동작이 같다면 테스트도 유지되어야 한다.
- query도 테스트 의도에 맞게 고른다. 단순 텍스트 포함인지, 특정 영역의 정확한 문구인지 구분한다.
- 외부에 드러난 interface가 props인지 화면인지 callback인지 먼저 정한다.

## 나쁜 신호
- 테스트가 컴포넌트 내부 파일 구조를 많이 알고 있다.
- 상태 관리 라이브러리를 바꾸면 테스트 대부분이 깨진다.
- 사용자에게 보이지 않는 값만 검증한다.
- 테스트명이 구현 설명에 가깝다.
- 테스트가 “어디에 무엇이 보여야 하는지”보다 “어떤 내부 함수가 호출됐는지”에 치우친다.
