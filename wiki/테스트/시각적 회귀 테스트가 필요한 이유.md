# 시각적 회귀 테스트가 필요한 이유

## 한 줄 요약
시각적 회귀 테스트는 단위 테스트가 잡기 어려운 의도하지 않은 UI 변화와 스타일 회귀를 스크린샷 비교로 검출한다.

## 핵심 내용
단위 테스트는 함수 호출, 상태, 텍스트, 속성을 잘 검증한다. 하지만 여백, 색상, 정렬, 잘림, 의존 컴포넌트의 스타일 변화는 놓칠 수 있다.

특히 디자인 시스템은 한 컴포넌트가 여러 제품과 컴포넌트에 사용되므로 작은 스타일 변경도 넓게 퍼질 수 있다.

우아한형제들 원문 사례는 이 문제를 잘 보여준다. 디자인 시스템이 115개 UI 컴포넌트를 제공하고 컴포넌트 단위 테스트 커버리지 91%를 유지해도, 스타일 코드 변경이나 의존 컴포넌트 변경으로 발생한 시각적 차이는 단위 테스트만으로 잡기 어렵다.

## 적합한 경우
- 디자인 시스템 컴포넌트
- 공통 UI 라이브러리
- 핵심 랜딩/결제/주문 화면
- 반복적으로 스타일 회귀가 발생한 영역
- QA 비용이 큰 화면

## 주의점
시각적 회귀 테스트는 모든 UI 테스트를 대체하지 않는다. 동작 정책은 단위/통합 테스트가 맡고, 시각 변화 감지는 스크린샷 테스트가 맡는다.

## 잡을 수 있는 문제
- 의도하지 않은 margin, padding, font, color 변경
- 하위 컴포넌트 변경으로 인한 상위 컴포넌트 UI 변화
- 공통 스타일 변경으로 인한 전체 컴포넌트 회귀
- 사람이 눈으로 확인하기 어려운 미세한 픽셀 차이
- MR에서 놓친 디자인 시스템 영향 범위
