# 테스트 코드의 비용과 효용

## 한 줄 요약
테스트는 많이 쓰는 것이 목표가 아니라, 작성·수정·실행 비용보다 장애 예방과 유지보수 효용이 큰 곳에 우선 배치하는 것이 목표다.

## 핵심 내용
테스트 코드도 코드다. 한 번 작성하고 방치하면 제품 변경과 함께 낡는다. 테스트도 기능 확장, 정책 변경, 버그 수정에 맞춰 계속 정리해야 한다.

테스트 우선순위는 다음 기준으로 정한다.

- 핵심 비즈니스 로직인가
- 장애가 나면 사용자 영향이 큰가
- 자주 변경되는 영역인가
- 수동 QA 비용이 큰가
- 회귀가 반복되는 영역인가

우아한형제들 Part 2의 핵심은 테스트도 “가성비”를 봐야 한다는 점이다. 테스트 작성 비용이 높아도 메인 비즈니스 로직이고 자주 변경되며 장애 위험이 크면 효용이 더 크다. 반대로 단순 표시 컴포넌트까지 복잡한 통합 테스트로 묶으면 비용이 효용을 넘을 수 있다.

## 실무 적용 기준
- 핵심 결제, 주문, 인증, 권한 흐름은 테스트 효용이 높다.
- 스타일만 있는 단순 wrapper는 테스트 효용이 낮을 수 있다.
- 너무 복잡한 테스트는 더 낮은 단위로 쪼갤 수 있는지 본다.
- 실패 원인을 알기 어려운 테스트는 유지보수 비용이 높다.
- 놓쳤던 케이스는 추가하고, 더 이상 의미 없는 케이스는 삭제한다.
- 테스트가 길고 복잡해도 핵심 로직이면 감수할 수 있다.
- 테스트 작성 여부를 오래 고민하는 것도 비용이다. 작고 확실한 테스트부터 둔다.

## 비용이 커지는 신호
- mock 준비가 assertion보다 길다.
- 실패했을 때 원인이 API mock, provider, 컴포넌트 중 어디인지 모른다.
- 기획 변경 때마다 테스트 대부분이 깨진다.
- 사용자 동작은 같은데 내부 구조 변경만으로 테스트가 깨진다.
