# 통합 테스트

> 여러 함수·훅·컴포넌트·API 응답이 연결될 때 **사용자 흐름**이 의도대로 동작하는지 검증.

## 경계 잡기
경계는 "몇 개 파일을 묶었는가"가 아니라 "사용자 관점의 의미 있는 흐름을 검증하는가"로 잡는다.

좋은 대상 — API 응답에 따라 UI가 바뀌는 화면 / 여러 컴포넌트 상태가 함께 움직이는 흐름 / 입력 후 버튼 활성화·가격 변경·목록 갱신이 이어지는 흐름 / 성공·오류에서 다른 UI.

우아한형제들 예시: 약관 API를 mock하고, 필수 약관을 체크하면 주문 버튼이 활성화되는 흐름. 단위 테스트가 `checkAllRequiredTerms()`와 `TermCheckBox`를 각각 보장해도, API + 여러 컴포넌트가 연결된 버튼 활성화까지 자동 보장하진 않는다.

**경계 잡는 질문** — 단일 함수 출력으로 충분한가? / API 응답과 UI가 맞물리는가? / 여러 컴포넌트가 상태를 공유·영향 주는가? / 전체 페이지를 렌더링하지 않아도 의미 있는 흐름을 검증할 수 있는가? 화면 전체를 무조건 렌더링하지 않고, 필요한 최소 경계만 묶는다. 외부 API는 [[MSW 통합 테스트|MSW]]로 제어.

## 컴포넌트와 API를 함께 테스트하기
API를 호출하는 컴포넌트는 조회·loading·empty·error·mutation 후 갱신 상태를 가져 props 테스트만으로 부족하다. MSW로 요청 계층을 가로채면 실제 API client 코드는 유지한 채 응답만 통제한다.

흐름:
1. 필요한 API handler 준비
2. 컴포넌트 렌더링
3. loading 종료·특정 데이터 노출을 기다림
4. 사용자 행동 수행
5. 화면 결과·버튼 상태·메시지·목록 변화 검증

검증 예 — 필수 약관 체크 시 주문 버튼 활성화 / 검색어 Enter 후 결과 노출 / 쿠폰 적용 후 포인트 금액 변경 / 404면 빈 상태 안내.

**주의** — API 함수 자체 mock에만 의존하면 실제 요청 경로와 멀어짐 / fixture가 정책과 어긋나면 신뢰도 하락 / 모든 서버 상태를 한 테스트에 몰면 유지보수 비용 증가.

## 사용자 시나리오 기반
사용자가 보는 화면 → 행동 → 바뀌는 결과를 하나의 흐름으로 검증. 단위 테스트의 Given-When-Then에 API 응답·여러 컴포넌트 연동이 추가된 형태.

- 테스트명은 사용자 행동 + 기대 결과를 함께 담는다.
- `userEvent` + 화면 기준 query(`getByRole`, `findByText`) 우선.
- 한 테스트는 하나의 의미 있는 흐름만. 너무 길면 정책 검증은 단위로 내리고 흐름만 남긴다.
- **나쁜 신호** — 테스트명이 "정상 작동한다" / 사용자 행동 없이 상태 직접 변경 / 여러 시나리오가 한 테스트에 섞임 / 성공·빈·오류가 한 케이스에 섞임.

## 단위 vs 통합 선택 기준
| 단위 테스트 | 통합 테스트 |
|---|---|
| 순수 함수 입출력 정책 | API 응답과 UI가 함께 동작 |
| hook의 독립 상태 전이 | 여러 컴포넌트 상태가 연결 |
| 컴포넌트의 작은 렌더링 조건 | 사용자 행동 후 화면 흐름이 이어짐 |
| 실패 원인을 빠르게 좁힐 핵심 로직 | 단위는 통과하나 화면 조합이 깨질 수 있음 |

선택 예 — 필수 약관 모두 체크됐는지 계산: 단위 / 체크박스 클릭 시 checked·onChange: 단위 또는 작은 컴포넌트 / API 응답 후 약관 체크 시 버튼 활성화: 통합 / 쿠폰 적용 후 포인트 정책: 정책 함수는 단위, 사용자 흐름은 통합.

**비용 기준** — 모든 동작을 고수준 테스트로 올리면 비효율. 대상이 커질수록 외부 라이브러리·API·provider mock이 늘고 실행 시간이 길어진다. 테스트 피라미드 관점으로 작은 테스트를 기본, 사용자 경험을 보장해야 하는 흐름만 통합으로 올린다.
