# 프론트엔드 통합 테스트의 경계

## 한 줄 요약
프론트엔드 통합 테스트는 여러 함수, 훅, 컴포넌트, API 응답이 함께 연결될 때 사용자 흐름이 의도대로 동작하는지 검증한다.

## 핵심 내용
통합 테스트의 경계는 “몇 개 파일을 묶었는가”보다 “사용자 관점의 의미 있는 흐름을 검증하는가”로 잡는다.

좋은 통합 테스트 대상:
- API 응답에 따라 UI 상태가 바뀌는 화면
- 여러 컴포넌트 상태가 함께 움직이는 흐름
- 사용자 입력 이후 버튼 활성화, 가격 변경, 목록 갱신이 이어지는 흐름
- 성공과 오류 응답에서 다른 UI를 보여주는 흐름

우아한형제들 통합 테스트 글은 약관 API 응답을 mock하고, 사용자가 필수 약관을 체크했을 때 주문 버튼이 활성화되는 흐름을 보여준다. 단위 테스트가 `checkAllRequiredTerms()`와 `TermCheckBox` 각각을 보장해도, API 응답과 여러 컴포넌트가 연결된 주문 버튼 활성화까지 자동으로 보장하지는 않는다.

## 실무 적용 기준
- 화면 전체를 무조건 렌더링하지 않는다.
- 검증하려는 사용자 흐름에 필요한 최소 경계만 묶는다.
- 외부 API는 MSW 같은 도구로 제어한다.
- 테스트가 너무 커지면 단위 테스트와 통합 테스트로 분리한다.
- 경계가 커질수록 mock 수, 실행 시간, 실패 원인 탐색 비용이 증가한다.
- 결제, 주문, 쿠폰, 포인트처럼 상호작용이 많은 영역은 통합 테스트 효용이 높다.

## 경계 잡는 질문
- 이 흐름은 단일 함수 출력으로 충분히 검증되는가?
- API 응답과 UI가 함께 맞물리는가?
- 여러 컴포넌트가 같은 상태를 공유하거나 서로 영향을 주는가?
- 전체 페이지를 렌더링하지 않아도 의미 있는 흐름을 검증할 수 있는가?
