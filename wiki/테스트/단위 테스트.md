# 단위 테스트

> 함수·훅·컴포넌트의 작은 단위가 **외부에서 기대한 동작**을 하는지 검증. 내부 구현이 아니라 공개 계약·화면 결과를 본다.

## 목적과 FIRST 원칙
단위 테스트는 내부 구현을 증명하는 코드가 아니다. 사용자가 접근 가능한 결과, 함수의 공개 계약, 컴포넌트의 렌더링·상호작용 결과를 검증한다.

좋은 단위 테스트의 역할:
- 변경 후 기존 동작이 깨졌는지 빠르게 알려준다.
- 코드가 만족해야 할 정책을 명세로 남긴다.
- 리팩터링의 안전망이 된다.
- 작은 단위의 실패 원인을 빠르게 좁힌다.

기본 특성은 F.I.R.S.T.
- **Fast** 빠르게 실행 / **Isolated** 외부 요인에 비종속 / **Repeatable** 반복해도 같은 결과 / **Self-validating** 성공·실패를 스스로 판단 / **Timely·Thorough** 적절한 시점 + 정상·비정상 흐름 충분히.

## 내부 구현을 테스트하면 안 되는 이유
내부 구현을 검증하면 사용자 동작은 그대로인데 테스트만 깨지고, 리팩터링을 방해한다.

- 버튼 클릭 후 Zustand store 값을 직접 확인하는 테스트는 컴포넌트가 아니라 상태 관리 라이브러리를 검증한다. 더 좋은 방식은 클릭 후 heading 문구가 의도대로 바뀌는지 보는 것.
- "바나나가 화면 어딘가 있다"도 약하다. 목적이 heading 변경이면 `heading` role + 정확한 accessible name으로 검증한다. 범위가 흐리면 우연히 통과한다.

**나쁜 신호** — 테스트가 내부 파일 구조를 많이 안다 / 상태 라이브러리를 바꾸면 테스트 대부분이 깨진다 / 안 보이는 값만 검증 / 테스트명이 구현 설명에 가깝다.

## React 컴포넌트 테스트 작성법
React Testing Library의 방향은 구현 세부가 아니라 사용자가 화면을 쓰는 방식에 가깝게 테스트하는 것.

우선순위:
1. `screen.getByRole` + accessible name으로 요소를 찾는다.
2. `userEvent`로 실제 사용자 행동에 가까운 이벤트를 발생.
3. 화면에 보이는 텍스트·role·상태를 검증.
4. 비동기 렌더링은 `findBy*`, `waitFor`로 기다림.

**Given-When-Then 흐름** — Given: props·mock 응답·초기 상태 준비 / When: `userEvent`로 사용자 행동 / Then: 화면·callback·접근성 상태 중 목적에 맞는 결과 검증. loading·empty·error도 별도 케이스. 비동기 UI는 loading 제거·데이터 노출·버튼 활성화 같은 관찰 가능한 지점을 기다린다.

**피해야 할 흐름** — state setter 직접 호출하고 결과를 건너뜀 / class name만으로 판단 / `waitFor` 안에 원인 모를 assertion 다수 / 사용자 행동 없이 구현 함수 직접 호출.

## 테스트 코드는 DAMP하게 작성한다
DAMP = Descriptive And Meaningful Phrases. 테스트는 구현 재사용(DRY)이 아니라 동작 명세 전달이 목적이라, 약간의 중복보다 "이 테스트가 무엇을 보장하는지"가 더 중요하다.

- 중요한 입력값은 테스트 안에서 명시한다. assertion 핵심 값도 보이게 둔다.
- Given-When-Then이 보이게 쓰고, 테스트 설명과 같은 이야기를 하게 한다.
- 공통 setup을 과하게 숨기면 시나리오가 사라진다. `renderWithEverything()` 하나로 조건이 다 가려지면 풀어쓴다.
- 균형점: 의미 없는 boilerplate는 helper로 줄이되, 의도를 설명하는 값·행위는 숨기지 않는다.

## 테스트 코드의 비용과 효용
테스트도 코드라 방치하면 낡는다. 많이 쓰는 게 목표가 아니라 **비용 < 효용**인 곳에 우선 배치한다.

우선순위 기준 — 핵심 비즈니스 로직인가 / 장애 시 사용자 영향이 큰가 / 자주 변경되는가 / 수동 QA 비용이 큰가 / 회귀가 반복되는가.

- 결제·주문·인증·권한 흐름은 효용이 높다. 스타일만 있는 단순 wrapper는 낮을 수 있다.
- 너무 복잡한 테스트는 더 낮은 단위로 쪼갤 수 있는지 본다. 놓친 케이스는 추가, 의미 없는 케이스는 삭제.
- **비용이 커지는 신호** — mock 준비가 assertion보다 길다 / 실패 시 원인이 mock·provider·컴포넌트 중 어디인지 모른다 / 기획 변경마다 테스트 대부분이 깨진다 / 내부 구조 변경만으로 깨진다.
