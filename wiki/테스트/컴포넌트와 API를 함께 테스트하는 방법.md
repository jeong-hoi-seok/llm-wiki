# 컴포넌트와 API를 함께 테스트하는 방법

## 한 줄 요약
컴포넌트와 API를 함께 테스트할 때는 실제 서버 대신 제어 가능한 mock 응답을 두고, 사용자가 보는 UI 변화와 상호작용 결과를 검증한다.

## 핵심 내용
API를 호출하는 컴포넌트는 데이터 조회, loading, empty, error, mutation 이후 갱신 같은 상태를 가진다. 이 상태는 컴포넌트 단독 props 테스트만으로 충분히 검증하기 어렵다.

MSW를 쓰면 요청 계층을 가로채서 실제 API client 코드는 유지한 채 응답만 테스트에서 통제할 수 있다.

우아한형제들 통합 테스트는 약관 API, 포인트 API, 쿠폰 API처럼 서버 응답이 UI 상태와 연결되는 예시를 다룬다. 카카오엔터테인먼트 MSW 글은 검색 입력 후 API 호출 결과가 목록에 반영되는 흐름을 다룬다. 공통점은 테스트가 “컴포넌트가 렌더링된다”에서 끝나지 않고 “서버 응답을 받은 뒤 사용자가 볼 상태가 맞는가”까지 간다는 점이다.

## 실무 적용 흐름
1. 테스트에서 필요한 API handler를 준비한다.
2. 컴포넌트를 렌더링한다.
3. loading 종료나 특정 데이터 노출을 기다린다.
4. 사용자 행동을 수행한다.
5. 화면 결과, 버튼 상태, 메시지, 목록 변화를 검증한다.

## 검증 예시
- 필수 약관을 모두 체크하면 주문 버튼이 활성화된다.
- 검색어 입력 후 Enter를 누르면 검색 결과가 노출된다.
- 쿠폰 적용 후 사용 가능 포인트와 적용 포인트 금액이 바뀐다.
- 404 응답이면 빈 상태 안내가 노출된다.

## 주의점
- API 함수 자체 mock에만 의존하면 실제 요청 경로와 멀어진다.
- 응답 fixture가 제품 정책과 달라지면 테스트 신뢰도가 떨어진다.
- 모든 서버 상태를 한 테스트에 몰아넣으면 유지보수 비용이 커진다.
- API handler가 많아지는 테스트는 경계를 낮출 수 있는지 본다.
