# 서버 상태가 변경되는 시나리오 테스트

## 한 줄 요약
서버 응답이 테스트 중 바뀌는 시나리오는 `server.use(...)`로 런타임 handler를 교체해 조건부 렌더링과 후속 UI 변화를 검증한다.

## 핵심 내용
카카오엔터테인먼트 원문에서 다루는 핵심은 실제 서버 데이터를 바꾸는 CRUD보다, 테스트 런타임에 API 응답을 동적으로 바꾸는 방식이다. 예시는 검색 API 응답의 `type`을 `rect`에서 `rounded`로 바꿔 카드 이미지 스타일이 달라지는지 검증한다.

이 방식은 다음 상황에 유용하다.

- 같은 API가 다른 데이터 shape을 내려줄 때
- 응답 값에 따라 컴포넌트 variant가 바뀔 때
- 특정 테스트에서만 빈 응답이나 오류 응답이 필요할 때
- fixture 파일을 새로 만들지 않고 일부 값만 바꾸고 싶을 때

## 테스트 흐름
1. 기본 handler와 fixture를 준비한다.
2. 특정 테스트에서 `server.use(...)`로 응답을 override한다.
3. 사용자가 입력, 클릭, keydown 같은 행동을 한다.
4. API 응답에 따른 UI 변화를 기다린다.
5. class, text, role, fallback 등 외부 결과를 검증한다.

## 주의점
- 테스트 간 서버 상태가 공유되면 결과가 흔들린다.
- fixture 객체를 직접 mutate하면 다른 테스트에 영향을 줄 수 있다. 새 객체를 만들어 override한다.
- `afterEach(server.resetHandlers)`를 반드시 둔다.
- handler override가 테스트 본문을 지저분하게 만들면 helper로 감싸되, 시나리오 이름은 숨기지 않는다.
- 실제 mutation 흐름까지 검증할 때는 낙관적 업데이트, 실패 rollback, 후속 query 갱신을 별도로 설계한다. 이 부분은 원문을 확장한 실무 적용이다.
