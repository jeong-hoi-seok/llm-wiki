# 테스트 시나리오를 자산으로 관리하기

## 한 줄 요약
테스트 시나리오는 일회성 검증 코드가 아니라 제품이 가질 수 있는 상태와 사용자 흐름을 설명하는 재사용 자산이다.

## 핵심 내용
좋은 시나리오는 테스트, Storybook, QA, 문서에서 반복해서 쓰인다. 그래서 시나리오는 코드 안에 숨기지 말고 이름과 구조를 갖춰 관리해야 한다.

29CM 글은 Storybook을 컴포넌트 명세서처럼 사용한다. 작은 컴포넌트부터 페이지까지 Story로 저장하고, API를 호출하는 컴포넌트는 MSW로 응답 상태를 만든다. 카카오엔터테인먼트 글은 그 Story의 MSW handler를 Jest 테스트에서도 재사용하는 방법을 제시한다.

따라서 시나리오는 다음 세 영역을 연결하는 자산이 된다.

- Storybook에서 눈으로 확인하는 UI 상태
- 자동화 테스트에서 검증하는 상태
- PR 리뷰에서 공유하는 Preview 상태

## 관리 대상
- 성공 상태
- 빈 상태
- 오류 상태
- 권한 없음
- 긴 콘텐츠
- 느린 응답
- 서버 상태 변경
- 주요 사용자 흐름

## 실무 적용 기준
- 시나리오 이름은 제품 언어로 짓는다.
- fixture와 handler는 시나리오 단위로 찾을 수 있게 둔다.
- Storybook Story 이름과 테스트 케이스 이름을 가깝게 유지한다.
- 장애가 난 케이스는 재발 방지 시나리오로 추가한다.
- PR 설명이나 리뷰 코멘트에서 관련 Story URL을 함께 제공한다.
- 신규 입사자가 Storybook만 봐도 컴포넌트 상태를 이해할 수 있게 Story를 관리한다.
