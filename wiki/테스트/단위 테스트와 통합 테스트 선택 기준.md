# 단위 테스트와 통합 테스트 선택 기준

## 한 줄 요약
단위 테스트는 작은 정책을 빠르게 검증할 때, 통합 테스트는 여러 단위가 연결된 사용자 흐름을 검증할 때 선택한다.

## 선택 기준
단위 테스트가 적합한 경우:
- 순수 함수의 입력/출력 정책
- hook의 독립적인 상태 전이
- 컴포넌트의 작은 렌더링 조건
- 실패 원인을 빠르게 좁혀야 하는 핵심 로직

통합 테스트가 적합한 경우:
- API 응답과 UI가 함께 동작한다.
- 여러 컴포넌트 상태가 연결된다.
- 사용자 행동 이후 화면 흐름이 이어진다.
- 단위 테스트는 통과하지만 실제 화면 조합이 깨질 수 있다.

## 실무 적용 기준
- 가능한 작은 테스트로 정책을 검증한다.
- 사용자 신뢰가 필요한 핵심 흐름은 통합 테스트로 보강한다.
- 모든 케이스를 고수준 통합 테스트로 올리지 않는다.
- 통합 테스트가 복잡해지면 하위 단위 테스트로 일부 정책을 내린다.

## 선택 예시
- 약관 배열에서 필수 약관이 모두 체크되었는지 계산한다: 단위 테스트
- 약관 체크박스 클릭 시 checked 상태와 onChange가 동작한다: 단위 테스트 또는 작은 컴포넌트 테스트
- 약관 API 응답을 받고 필수 약관을 모두 체크하면 주문 버튼이 활성화된다: 통합 테스트
- 쿠폰 적용 후 포인트 적용 금액이 정책에 맞게 바뀐다: 정책 함수는 단위 테스트, 사용자 흐름은 통합 테스트

## 비용 기준
우아한형제들 통합 테스트 글은 모든 동작을 고수준 테스트로 검증하는 것이 비효율적이라고 설명한다. 테스트 대상이 커질수록 외부 라이브러리, API, provider mock이 늘고 실행 시간도 길어진다. 따라서 테스트 피라미드 관점으로 작은 테스트를 기본으로 두고, 실제 사용자 경험을 보장해야 하는 흐름만 통합 테스트로 올린다.
