# 시각적 회귀 테스트

> 단위 테스트가 놓치는 **의도하지 않은 UI 변화·스타일 회귀**를 스크린샷 비교로 검출. 동작 정책은 단위·통합이, 시각 변화는 스크린샷이 맡는다.

## 왜 필요한가
단위 테스트는 함수·상태·텍스트·속성을 잘 검증하지만 여백·색상·정렬·잘림, 의존 컴포넌트의 스타일 변화는 놓친다. 특히 디자인 시스템은 한 컴포넌트가 여러 제품에 쓰여 작은 변경도 넓게 퍼진다.

우아한형제들 사례: 디자인 시스템이 115개 UI 컴포넌트를 제공하고 단위 테스트 커버리지 91%를 유지해도, 스타일 코드·의존 컴포넌트 변경으로 생긴 시각 차이는 단위 테스트로 잡기 어렵다.

잡을 수 있는 문제 — 의도치 않은 margin·padding·font·color 변경 / 하위 컴포넌트 변경으로 인한 상위 UI 변화 / 공통 스타일 변경으로 인한 전체 회귀 / 미세한 픽셀 차이 / MR에서 놓친 디자인 시스템 영향 범위.

적합한 경우 — 디자인 시스템 컴포넌트 / 공통 UI 라이브러리 / 핵심 랜딩·결제·주문 화면 / 스타일 회귀가 반복된 영역 / QA 비용이 큰 화면. 모든 UI 테스트를 대체하진 않는다.

## Playwright 스크린샷 테스트
Playwright는 브라우저에서 렌더링·스크린샷을 찍어 expected·actual·diff를 비교한다. 실패 시 HTML 리포트·trace로 어디가 달라졌는지 확인. 우아한형제들이 Playwright를 택한 이유 — 무료, 한글 지원, E2E 확장성, HTML 리포트(Chromatic은 유료, BackstopJS는 한글 문제로 제외).

흐름: 대상 Storybook·페이지 열기 → 렌더링 완료 대기 → 스크린샷 → 기준 이미지와 비교 → 의도한 변경이면 snapshot 갱신, 아니면 UI 수정.

결과물 — expected(기준) / actual(현재) / diff(차이) / HTML report(실패 원인·단계·mismatch·trace).

적용 — CI 같은 고정 환경에서 실행 / 브라우저·OS·font 차이 통제 / snapshot 갱신은 수동 승인 / diff 리포트를 MR/PR에서 바로 / 신규 컴포넌트는 기준 이미지가 없어 실패하니 생성 이미지를 기준으로 등록하는 흐름 필요.

## 디자인 시스템 회귀
공통 컴포넌트(버튼·입력·팝오버·모달) 변경이 여러 제품 화면으로 퍼진다. 우아한형제들은 테스트베드로 Storybook을 택했다 — iframe 경로로 컴포넌트만 볼 수 있고 상태별 Story가 이미 있어 추가 코드가 적었다.

회귀 테스트 층:
- 단위: props·이벤트·접근성 속성
- 스토리: 상태별 렌더링
- 시각적 회귀: 스타일·레이아웃 변화
- E2E: 실제 사용자 흐름

- 컴포넌트별 주요 variant + disabled·error·loading·long text 상태를 Story로.
- Story가 곧 회귀 대상이라 Story 품질이 낮으면 테스트 품질도 낮다. 디자인 QA용 Story와 자동화용 Story가 멀어지지 않게.

## 변동성 제어 (flaky 방지)
스크린샷은 작은 차이에도 실패한다. 민감함은 장점이지만 환경이 흔들리면 flaky가 된다. 우아한형제들의 두 대표 문제 — 렌더링 끝나기 전 촬영돼 빈 화면이 남음 / `setTimeout` 기반 자동 슬라이드가 실행마다 다른 화면. CSR에선 `load`·`domContentLoaded`만으론 React 렌더 완료를 보장 못 해, MutationObserver로 DOM 변화가 멈출 때까지 기다림.

통제 대상 — 렌더링 완료 전 촬영 / 타이머·자동 슬라이드 / 애니메이션·transition / 외부 폰트 로딩 / 랜덤 값·현재 시간 / OS·브라우저·font rendering 차이 / 네트워크 지연·이미지 로딩.

- CI 고정 Docker image에서 실행(로컬보다 환경 차이 적음). 애니메이션은 테스트 환경에서 끔. 시간·랜덤 고정.
- 렌더링 완료 신호를 기다린 뒤 촬영. snapshot update는 수동 job으로 분리. 리포트는 MR/PR에서 바로 열기.
