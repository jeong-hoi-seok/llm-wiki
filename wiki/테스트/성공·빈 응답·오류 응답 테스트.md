# 성공·빈 응답·오류 응답 테스트

## 한 줄 요약
API 기반 화면은 성공 응답뿐 아니라 빈 응답, 엣지 케이스, 오류 응답에서 사용자가 보는 UI를 함께 검증해야 한다.

## 핵심 내용
프론트엔드 장애는 성공 흐름보다 비정상 응답에서 자주 드러난다. 데이터가 없거나, 필드가 비어 있거나, 권한이 없거나, 서버가 실패할 때 UI가 무너지지 않아야 한다.

카카오엔터테인먼트 원문은 검색 결과 화면을 기준으로 네 가지 흐름을 검증한다.

- 서버에서 데이터를 정상적으로 가져와 렌더링하는가
- 서로 다른 데이터에 따라 조건부 렌더링이 되는가
- 빈 리스트 같은 엣지 케이스를 처리하는가
- 네트워크 에러가 발생해도 fallback이 동작하는가

## 테스트해야 할 상태
- loading: 요청 중 안내가 보이는가
- success: 데이터가 정상 렌더링되는가
- empty: 빈 상태 메시지와 다음 행동이 보이는가
- error: 오류 메시지와 재시도 행동이 보이는가
- edge: 값 누락, 긴 텍스트, 비정상 수량을 처리하는가

## 실무 적용 기준
- handler를 상태별로 분리한다.
- empty와 error를 success 테스트 안에 섞지 않는다.
- 사용자에게 보이는 복구 경로를 검증한다.
- 에러 로그보다 화면 피드백을 우선 검증한다.
- 404, 403 같은 상태 코드는 서버와 합의한 정책에 맞춰 테스트한다.
- 공통 interceptor를 쓴다면 error handler 호출 여부도 별도 테스트한다.

## 예시 판단
검색 결과가 없을 때 빈 배열이 내려오는 경우와 404가 내려오는 경우는 다르다. 둘 다 같은 “검색 결과가 없습니다” UI를 보여줄 수 있지만, 테스트에서는 원인이 다른 케이스로 분리하는 편이 안전하다.

## 나쁜 신호
- 성공 응답만 테스트한다.
- 빈 응답을 fixture 하나로만 눈으로 확인하고 자동화하지 않는다.
- 에러 응답에서 앱이 죽지 않는지만 보고 사용자가 볼 메시지는 검증하지 않는다.
- 테스트 실패 시 API mock 문제인지 UI 문제인지 구분하기 어렵다.
