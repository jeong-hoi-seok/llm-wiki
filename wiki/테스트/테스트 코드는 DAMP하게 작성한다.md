# 테스트 코드는 DAMP하게 작성한다

## 한 줄 요약
테스트 코드는 중복 제거보다 읽는 사람이 시나리오와 검증 의도를 바로 이해할 수 있는 설명력을 우선해야 한다.

## 핵심 내용
DAMP는 Descriptive And Meaningful Phrases의 약자다. 테스트 코드에서는 DRY보다 DAMP가 더 중요할 때가 많다.

테스트의 목적은 구현 재사용이 아니라 동작 명세 전달이다. 공통 setup을 과하게 숨기면 테스트를 읽는 사람이 어떤 조건에서 어떤 행동을 했고 무엇을 기대하는지 파악하기 어렵다.

원문은 같은 기능을 검증하는 두 테스트를 비교하며, 미리 정의된 props와 helper 뒤에 중요한 값이 숨어 있으면 테스트가 통과하더라도 읽는 사람이 시나리오를 바로 이해하기 어렵다고 설명한다. 테스트에서는 약간의 중복보다 “이 테스트가 무엇을 보장하는지”가 더 중요하다.

## 실무 적용 기준
- 테스트 내부에서 중요한 입력값은 명시한다.
- Given-When-Then 흐름이 보이게 작성한다.
- 너무 많은 helper가 시나리오를 숨기면 풀어쓴다.
- 중복 제거는 “읽기 쉬움”을 해치지 않는 범위에서만 한다.
- assertion에 들어가는 핵심 값은 테스트 안에서 보이게 둔다.
- 테스트 설명과 Given-When-Then이 같은 이야기를 하게 만든다.

## 균형점
DAMP는 중복을 무한 허용하라는 뜻이 아니다. 의미 없는 boilerplate는 helper로 줄이되, 테스트 의도를 설명하는 값과 행위는 숨기지 않는다.

## 나쁜 신호
- `renderWithEverything()` 하나로 중요한 조건이 모두 가려진다.
- 테스트 제목을 읽어도 어떤 데이터로 검증하는지 모른다.
- helper를 따라가야만 사용자 행동을 이해할 수 있다.
