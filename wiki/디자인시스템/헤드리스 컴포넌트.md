# 헤드리스 컴포넌트

> 로직·접근성은 공통, 스타일은 위임. 상위: [[index]]

## TL;DR
`unstyled`를 넘어 **상태·인터랙션·접근성 같은 "동작(behavior)"만 제공**하고, **시각 표현(UI)은 사용처가 결정**하도록 위임하는 패턴. 가이드(일관성)와 자유(커스터마이징)를 양립시킨다.

| | 헤드리스가 제공 | 사용처가 결정 |
|---|---|---|
| 상태 관리 (open/selected…) | ✅ | |
| 키보드·포커스·ARIA ([[접근성]]) | ✅ | |
| 인터랙션 로직 | ✅ | |
| 색·여백·타이포·레이아웃 | | ✅ |

## 왜 중요한가
- 접근성 로직 — **브라우저별 차이, 키보드 포커스 이동, ARIA 속성** — 을 모든 컴포넌트마다 직접 구현하는 건 비현실적.
- 헤드리스로 **로직·접근성을 한 번만** 만들고 UI만 갈아끼우면 → 안정성은 공통, 디자인은 자유.
- 디자인 완전 균일화가 목표가 아닌 서비스에 특히 유효.

## 구조 (실무 적용)
```
Headless (로직 + 접근성만)
 ├─ 기본 UI 버전   — 디자인 가이드에 맞춘 스타일
 └─ 자유 UI 버전   — 서비스 특성에 맞춘 커스텀 스타일
```

## 헤드리스 vs 스타일드(Styled)
| | 헤드리스 | 스타일드 (예: MUI, Antd) |
|---|---|---|
| 스타일 | 없음 (위임) | 내장 (테마) |
| 커스터마이징 자유 | 높음 | 낮음~중간 (override 싸움) |
| 초기 속도 | 느림 (스타일 직접) | 빠름 (바로 예쁨) |
| 디자인 일관성 강제 | 약함 | 강함 |
| 적합 | 자체 디자인 시스템 | 빠른 프로토타입·내부 툴 |

## 대표 라이브러리
| 라이브러리 | 특징 |
|---|---|
| **Radix UI Primitives** | 가장 널리 쓰이는 헤드리스. 접근성 탄탄, [[shadcn-ui|shadcn/ui]] 기반 |
| **Headless UI** | Tailwind 팀 제작, React·Vue |
| **React Aria** (Adobe) | 접근성·국제화 최고 수준, 훅 기반 |
| **TanStack Table/Virtual** | 데이터·가상화 로직 헤드리스 |
| **Reach UI** | 초기 대표주자 (현재 유지보수 둔화) |

## 실전 사례 — shadcn/ui
**[[shadcn-ui|shadcn/ui]]** = 헤드리스(Radix) + Tailwind 스타일을 **복붙해서 소유**하는 모델. 위 "Headless → 기본 UI 버전" 구조의 대표 구현체.
- Radix Primitives(로직·접근성) 위에 Tailwind 스타일을 입힌 컴포넌트.
- npm 의존성이 아니라 **CLI로 코드를 프로젝트에 복사** → 코드를 소유하고 자유 수정 → "자유 UI 버전"으로 바로 확장.
- 자세히 → [[shadcn-ui]]

## 왜 [[공통 컴포넌트 추출 기준|공통 컴포넌트]] 문제의 해법인가
공통 컴포넌트가 비대화(책임 과다)하거나 UI 강제로 자유를 잃는 문제를 **로직/접근성은 공통화 + UI 결합은 사용처 위임**으로 분리 → [[결합도|결합도]]↓, 커스터마이징 자유↑.
