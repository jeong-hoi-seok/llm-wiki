# AI 티 안 나는 퍼블리싱 기준

## 한 줄 요약
디자인 시스템이나 명확한 시안 없이 AI가 디자인부터 퍼블리싱까지 맡을 때는 익숙한 SaaS 템플릿과 과한 장식을 자동 적용하지 말고, **제품 맥락·콘텐츠 성격·사용 흐름에 맞는 절제된 화면**을 만든다.

## 이 문서의 범위
이 문서는 미적 취향 강제가 아니라 AI가 습관적으로 만드는 “그럴듯하지만 어디서 본 듯한 화면”을 줄이기 위한 기준이다.

디자인 시스템, 브랜드 가이드, Figma 시안, 프로젝트 `docs/`가 있으면 그것이 우선이다. 그런 기준이 없을 때 이 문서를 기본 억제 규칙으로 사용한다.

## 핵심 원칙

### 1. 패턴보다 맥락을 먼저 본다
AI는 디자인 요청을 받으면 익숙한 SaaS 랜딩 패턴을 자동으로 적용하기 쉽다. 먼저 제품 성격을 판단한다.

| 확인할 것 | 질문 |
|---|---|
| 제품 유형 | SaaS, 커머스, 포트폴리오, 문서, 대시보드, 도구 중 무엇인가? |
| 사용자 목적 | 읽기, 탐색, 구매, 입력, 비교, 반복 작업 중 무엇인가? |
| 콘텐츠 밀도 | 짧은 홍보 문구인가, 상세 정보인가, 작업 데이터인가? |
| 브랜드 톤 | 차분함, 전문성, 실험성, 엔터프라이즈, 개인성 중 무엇이 맞는가? |
| 화면 성격 | 랜딩인가, 실제 앱 화면인가, 운영 도구인가? |

맥락이 불명확하면 장식을 늘리지 말고 정보 구조를 먼저 만든다.

### 2. 모든 것을 강조하지 않는다
제목, 카드, 배지, 버튼, 아이콘, 애니메이션을 모두 강하게 강조하면 화면 전체가 시끄럽고 AI 생성물처럼 보인다.

강조는 우선순위가 있을 때만 쓴다.

- 가장 중요한 행동 1개
- 현재 사용자가 봐야 할 상태 1개
- 비교가 필요한 핵심 차이
- 위험하거나 되돌리기 어려운 액션

나머지는 조용하게 둔다.

### 3. 카드와 장식은 의미가 있을 때만 쓴다
카드는 정보를 분리하거나 반복 항목을 묶을 때 쓴다. 모든 문단과 섹션을 카드로 감싸지 않는다.

카드를 쓰기 좋은 경우:
- 반복 리스트
- 요금제 비교
- 선택 가능한 항목
- 독립된 상태 요약
- 모달·패널·도구 영역

카드를 피할 경우:
- 단순 본문 섹션
- 페이지 전체를 감싸는 장식
- 카드 안의 카드
- 정보 구분보다 “예뻐 보이게” 하려는 목적

## 피해야 할 AI 디자인 습관

### 레이아웃
- 모든 콘텐츠를 습관적으로 중앙 정렬하지 않는다.
- `Hero → 3열 Feature Card → CTA`로 반복되는 SaaS 템플릿을 자동 적용하지 않는다.
- 콘텐츠 성격과 무관하게 카드를 나누지 않는다.
- 카드 안에 또 카드를 넣는 중첩 구조를 피한다.
- 제품 맥락 대신 익숙한 SaaS 디자인 패턴을 자동 적용하지 않는다.

### 시각 스타일
- 모든 컨테이너에 큰 radius, border, shadow를 반복하지 않는다.
- 제품 브랜드와 무관한 보라색·파란색 gradient를 기본값처럼 쓰지 않는다.
- 제목 주변에 색상 선, 막대, 점을 반복해서 붙이지 않는다.
- 모든 제목 위에 eyebrow, badge, overline을 붙이지 않는다.
- gradient orb, glow, blur blob, glassmorphism 같은 의미 없는 장식을 쓰지 않는다.
- 제목, 카드, 배지, 버튼을 모두 강하게 강조하지 않는다.

### 아이콘
- 모든 기능 항목에 반복적인 아이콘 타일을 붙이지 않는다.
- Lucide 같은 범용 아이콘을 모든 제목, 버튼, 카드에 습관적으로 붙이지 않는다.
- 아이콘은 의미 전달이나 탐색을 실제로 돕는 경우에만 쓴다.

아이콘을 써도 되는 경우:
- 삭제, 저장, 다운로드, 검색, 닫기처럼 명령 의미가 명확한 버튼
- 네비게이션에서 반복적으로 인지해야 하는 항목
- 텍스트보다 아이콘이 더 빠르게 이해되는 도구
- 상태를 구분해야 하는 데이터 UI

### 애니메이션
- 모든 요소에 동일한 fade-in, slide-up, hover lift를 적용하지 않는다.
- hover 애니메이션은 클릭 가능성이나 상태 변화를 알려줄 때만 쓴다.
- 페이지 진입 애니메이션보다 정보 배치와 읽기 흐름을 우선한다.

## 좋은 기본값

### 정보형 페이지
- 좌측 정렬을 기본으로 한다.
- 본문 폭을 제한한다.
- 제목, 요약, 섹션 제목, 본문, 보조 정보 위계만으로 충분한지 먼저 본다.
- 카드보다 여백, 구분선, 타이포그래피로 구조를 만든다.

### 실제 앱·도구 화면
- 랜딩 페이지처럼 만들지 않는다.
- 작업 대상, 현재 상태, 주요 액션, 필터/검색, 결과 영역이 바로 보이게 한다.
- 반복 사용자가 스캔하기 쉽게 밀도와 정렬을 조절한다.
- 장식보다 상태, 오류, 비활성, 로딩, 빈 상태를 챙긴다.

### 대시보드·운영 화면
- 큰 hero를 만들지 않는다.
- 카드 남발보다 표, 리스트, 필터, 탭, 사이드바 같은 작업 구조를 우선한다.
- 색은 상태 구분과 위험도 표현에 주로 쓴다.

### 랜딩 페이지
- Hero는 제품/브랜드/제안이 첫 화면에서 바로 드러나야 한다.
- 3열 feature card를 기본값으로 쓰지 말고, 제품이 실제로 어떻게 작동하는지 보여준다.
- CTA는 핵심 행동 위주로 제한한다.
- 의미 없는 gradient 배경보다 제품 이미지, 실제 화면, 구체적 사례가 낫다.

## 에이전트 체크리스트

작업 전:
- [ ] 디자인 시스템이나 Figma 시안이 있는가?
- [ ] 프로젝트 `docs/`에 브랜드·UI 규칙이 있는가?
- [ ] 화면이 랜딩인지 실제 앱인지 구분했는가?
- [ ] 사용자가 이 화면에서 하려는 일이 무엇인지 정했는가?

구현 중:
- [ ] 모든 섹션을 카드로 만들고 있지 않은가?
- [ ] 모든 제목에 badge/eyebrow/아이콘을 붙이고 있지 않은가?
- [ ] 브랜드와 무관한 gradient를 기본값으로 쓰고 있지 않은가?
- [ ] hover lift나 fade-in을 모든 요소에 반복 적용하지 않았는가?
- [ ] 아이콘이 의미 전달 없이 장식으로만 쓰이고 있지 않은가?

마무리:
- [ ] 한 화면에서 가장 강한 요소가 너무 많지 않은가?
- [ ] 좌측 정렬이 더 적합한 정보를 중앙 정렬하지 않았는가?
- [ ] 카드 없이도 충분한 곳에 카드를 쓰지 않았는가?
- [ ] 디자인보다 콘텐츠와 작업 흐름이 먼저 보이는가?
- [ ] “익숙한 SaaS 템플릿”처럼 보이는 부분이 없는가?
