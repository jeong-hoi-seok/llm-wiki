# 모듈러 아키텍처

## 한 줄 요약
모듈러 아키텍처는 함께 바뀌는 코드를 모듈로 묶고 모듈 사이 접점을 작게 만들어, **기능 변경이 다른 영역으로 번지는 것을 줄이는 구조화 방식**이다. 상위: [[프론트엔드 아키텍처 개요]]

## 이 문서의 범위
이 문서는 프론트엔드 모듈화의 기본 원칙을 설명한다. 특정 폴더명, 특정 프레임워크, 특정 아키텍처 패턴을 강제하지 않는다.

프로젝트에서 실제 모듈 단위와 public API 규칙은 프로젝트 `docs/`에서 결정한다.

## 핵심 개념
모듈은 코드 묶음이 아니라 책임의 경계다. 좋은 모듈은 내부 응집도가 높고 외부 결합도가 낮다.

| 기준 | 좋은 모듈 | 나쁜 모듈 |
|---|---|---|
| 변경 이유 | 같은 이유로 함께 바뀜 | 서로 다른 이유로 자주 바뀜 |
| 외부 노출 | 필요한 API만 공개 | 내부 파일을 외부에서 직접 참조 |
| 의존성 | 적은 수의 명확한 의존 | 여러 방향으로 얽힌 의존 |
| 이름 | 도메인·기능 의미가 드러남 | `utils`, `common`, `components` 남용 |
| 테스트 | 독립 검증 가능 | 다른 화면 전체가 있어야 검증 가능 |

## 왜 중요한가
- 기능 추가 시 수정 위치를 예측하기 쉬워진다.
- 같은 기능의 UI, 상태, API, 검증 로직을 함께 이해할 수 있다.
- 내부 구현을 바꿔도 외부 사용 코드를 덜 깨뜨린다.
- 중복 제거보다 변경 범위 통제가 우선이라는 기준을 준다.

## 역할 기반 분리와 기능 기반 분리
역할 기반 분리는 기술 종류별로 파일을 모은다.

```txt
components/
hooks/
utils/
api/
types/
```

작은 프로젝트에서는 단순하다. 하지만 기능이 커지면 한 기능을 이해하기 위해 여러 폴더를 오가야 한다.

기능 기반 분리는 같은 기능이나 도메인과 관련된 코드를 가깝게 둔다.

```txt
payment/
  ui/
  api/
  model/

profile/
  ui/
  api/
  model/
```

기능 기반 분리는 응집도를 높인다. 단, 공통 기반 코드와 도메인 코드를 섞지 않도록 레이어 규칙이 함께 필요하다.

## 모듈 경계 잡는 질문
| 질문 | 의미 |
|---|---|
| 이 코드들은 같은 기능을 위해 함께 읽히는가? | 모듈 후보 |
| 이 코드들은 같은 변경 요청에서 함께 수정되는가? | 같은 모듈 후보 |
| 이 코드가 특정 도메인 용어를 포함하는가? | 도메인/기능 모듈 후보 |
| 이 코드가 여러 도메인에서 의미 없이 재사용되는가? | 공통 기반 후보 |
| 외부에서 내부 파일을 직접 import해야 하는가? | public API 부족 가능성 |

## Public API
모듈은 내부 구조를 숨기고 외부에는 필요한 진입점만 제공하는 편이 좋다.

```txt
좋음:
features/login/index.ts
features/login/ui/LoginForm.tsx
features/login/model/useLogin.ts

외부 사용:
import { LoginForm } from '@/features/login';
```

내부 파일 직접 import가 많아지면 모듈 내부 구조를 바꾸기 어려워진다.

```txt
나쁨:
import { LoginForm } from '@/features/login/ui/LoginForm';
import { useLogin } from '@/features/login/model/useLogin';
```

단, public API 강제 여부와 파일명 규칙은 프로젝트 `docs/`에서 정한다.

## 추출 기준
모듈 추출은 코드 줄 수보다 변경 이유와 재사용 범위로 판단한다.

| 상황 | 판단 |
|---|---|
| 같은 화면 안에서만 쓰이고 독립 의미가 약함 | 굳이 분리하지 않는다. |
| 같은 기능의 상태·검증·UI가 반복됨 | 기능 모듈 후보. |
| 여러 도메인에서 같은 UI 패턴을 씀 | 공통 UI 후보. |
| 도메인 정책이 여러 화면에 흩어짐 | 도메인 모듈 후보. |
| 중복은 있지만 변경 이유가 다름 | 섣불리 공통화하지 않는다. |

공통화 판단은 [[공통 컴포넌트 추출 기준]]를 함께 본다.

## 에이전트 적용 기준
- 먼저 프로젝트 `docs/`에서 모듈 기준과 import 규칙을 확인한다.
- 프로젝트 규칙이 없으면 기능/도메인 단위 응집도를 우선한다.
- `utils`, `common`, `components`가 커지면 역할 기반 분리 과잉 가능성을 점검한다.
- 내부 파일 직접 import가 반복되면 public API 필요성을 검토한다.
- Obsidian 문서는 판단 기준이며, 실제 폴더명 확정은 프로젝트 `docs/`가 담당한다.

## 안티패턴
- 중복 제거만 보고 서로 다른 도메인 코드를 하나로 합친다.
- 공통 폴더가 도메인 규칙을 흡수한다.
- 모듈 외부에서 내부 구현 파일을 자유롭게 import한다.
- 기능 하나를 이해하려면 프로젝트 전체 폴더를 돌아다녀야 한다.
- 폴더 구조만 기능 기반이고 의존성은 아무 방향 없이 얽힌다.
