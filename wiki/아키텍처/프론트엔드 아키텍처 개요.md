# 프론트엔드 아키텍처 개요

## 한 줄 요약
프론트엔드 아키텍처는 UI, 상태, 데이터 흐름, 도메인 로직, 외부 의존성을 적절한 경계로 나누어 **변경 영향 범위를 작게 만들고 예측 가능하게 유지하는 설계 방식**이다. 상위: [[index]]

## 이 문서의 범위
이 문서는 특정 프로젝트 규칙이 아니다. 어떤 프로젝트에도 적용할 수 있는 프론트엔드 아키텍처의 기본 목적과 판단 기준을 정리한다.

프로젝트가 실제로 어떤 폴더 구조, import 규칙, 상태 관리 도구, 데이터 fetching 전략을 채택하는지는 각 프로젝트의 `docs/`에서 결정한다.

## 핵심 개념
프론트엔드 코드는 보통 다음 관심사가 섞이기 쉽다.

| 관심사 | 설명 | 섞였을 때 문제 |
|---|---|---|
| UI | 화면 구조, 스타일, 사용자 입력 | 작은 UI 수정이 비즈니스 로직에 영향 |
| 상태 | 서버 상태, 클라이언트 상태, URL 상태, 폼 상태 | 데이터 출처와 변경 책임이 불명확 |
| 데이터 통신 | API 요청, 캐싱, 에러, 재시도 | 화면마다 중복 처리 |
| 도메인 규칙 | 서비스 정책, 검증, 계산 | UI 변경과 정책 변경이 함께 얽힘 |
| 플랫폼 의존성 | 라우터, 스토리지, 네이티브 API, 브라우저 API | 테스트와 재사용 어려움 |

아키텍처는 이 관심사들을 무조건 많이 나누는 일이 아니다. 변경 이유가 다른 코드가 서로 끌려다니지 않도록 경계를 만드는 일이다.

## 왜 중요한가
- **변경 영향 범위 축소**: 수정한 코드가 어디까지 영향을 주는지 예측하기 쉬워진다.
- **온보딩 비용 감소**: 새 팀원이 파일 위치와 의존 방향을 빠르게 이해한다.
- **중복 판단 일관화**: 공통화, 분리, 추출 기준이 코드마다 달라지지 않는다.
- **테스트 가능성 향상**: UI와 로직, 데이터 통신을 분리하면 작은 단위 검증이 쉬워진다.
- **장기 유지보수성 향상**: 프로젝트가 커져도 기능 추가와 리팩터링 비용이 급격히 커지는 것을 늦춘다.

## 기본 판단 기준
프론트엔드 아키텍처를 볼 때 먼저 아래 질문을 던진다.

1. 이 코드는 어떤 이유로 변경되는가?
2. 이 코드는 어디까지 재사용되는가?
3. 이 코드는 어떤 레이어나 모듈을 알아야 하는가?
4. 이 코드는 UI, 상태, 데이터 통신, 도메인 규칙 중 무엇을 책임지는가?
5. 이 코드를 바꾸면 어떤 화면이나 기능이 함께 깨질 수 있는가?

답이 흐리면 구조가 부족하거나 책임이 섞였을 가능성이 높다.

## 패턴과 규칙의 차이
아키텍처 패턴은 선택지다. 프로젝트 규칙은 채택 결과다.

| 구분 | 위치 | 예 |
|---|---|---|
| 개념·원칙 | Obsidian wiki | 레이어, 모듈, 응집도, 결합도 |
| 선택 가능한 패턴 | Obsidian wiki | [[FSD 폴더구조]] |
| 프로젝트 채택 결정 | 프로젝트 `docs/` | "이 프로젝트는 FSD를 채택한다" |
| 실행 규칙 | 프로젝트 `AGENTS.md`, `docs/` | import 금지 규칙, 폴더 생성 규칙 |

Obsidian은 프로젝트 `docs/`의 문서 품질을 높이는 참고 지식이다. Obsidian 문서만으로 특정 프로젝트의 강제 규칙을 판단하지 않는다.

## 에이전트 적용 기준
- 항상 적용: 관심사 분리, 변경 영향 범위 축소, 응집도·결합도 판단.
- 참고로 적용: 레이어드 구조, 모듈 구조, FSD 같은 패턴.
- 프로젝트별 확인 필요: 실제 폴더명, alias, import 제한, 상태 관리 도구, API 배치 규칙.
- 프로젝트 `docs/`와 Obsidian 설명이 다르면 프로젝트 `docs/`를 우선한다.

## 안티패턴
- 폴더명만 정하고 의존 방향은 정하지 않는다.
- 모든 공통 코드를 `shared`, `common`, `utils`에 넣는다.
- 화면 컴포넌트가 API 요청, 도메인 계산, 전역 상태 변경을 모두 직접 처리한다.
- 특정 패턴을 프로젝트 상황과 무관하게 정답처럼 적용한다.
- Obsidian의 패턴 문서를 프로젝트 채택 규칙으로 오해한다.
