# 레이어드 아키텍처

## 한 줄 요약
레이어드 아키텍처는 코드를 추상화 수준과 책임에 따라 층으로 나누고, **의존 방향을 한쪽으로 제한해 변경 전파를 줄이는 구조화 방식**이다. 상위: [[프론트엔드 아키텍처 개요]]

## 이 문서의 범위
이 문서는 일반적인 레이어 사고를 설명한다. 특정 폴더명이나 FSD 채택을 강제하지 않는다.

프로젝트에서 실제 레이어 이름과 import 규칙은 프로젝트 `docs/`에서 결정한다. [[FSD 폴더구조]]은 레이어드 사고를 구체화한 패턴 중 하나다.

## 핵심 개념
레이어는 코드가 맡는 책임과 재사용 범위를 기준으로 나눈다.

| 레이어 관점 | 맡는 책임 | 예 |
|---|---|---|
| 앱 조립 | 라우팅, 프로바이더, 전역 설정 | app entry, router, theme provider |
| 화면 | 페이지 단위 데이터 연결과 화면 구성 | page, screen |
| 기능 | 사용자가 수행하는 행동 | login, add-to-cart, submit-form |
| 도메인 | 서비스가 다루는 핵심 데이터와 규칙 | user, product, order |
| 공통 기반 | 비즈니스와 무관한 재사용 도구 | button, date, http client |

좋은 레이어 구조는 상위가 하위를 사용하고, 하위는 상위를 모른다.

```txt
app/screen
  -> feature
    -> domain/entity
      -> shared/base
```

## 왜 중요한가
- 하위 레이어가 상위 화면을 모르면 재사용 가능성이 높아진다.
- 비즈니스 규칙이 UI 세부사항에 묶이지 않는다.
- import 방향이 단순해져 순환 의존을 줄인다.
- 변경 영향 범위가 위쪽으로만 흐르기 때문에 디버깅이 쉬워진다.

## 의존 방향
레이어드 아키텍처의 핵심은 폴더 이름이 아니라 의존 방향이다.

```txt
허용: 상위 레이어 -> 하위 레이어
금지: 하위 레이어 -> 상위 레이어
주의: 같은 레이어 내부의 무분별한 직접 의존
```

예시:

```txt
page가 shared button을 사용한다        OK
shared button이 page 상태를 import한다  NG
domain 모델이 router를 import한다       NG
feature가 domain 타입을 사용한다        OK
```

## 레이어를 나누는 기준
| 질문 | 판단 |
|---|---|
| 이 코드는 특정 화면에만 필요한가? | 화면 가까이에 둔다. |
| 여러 화면에서 같은 사용자 행동으로 재사용되는가? | 기능 레이어 후보. |
| 서비스 핵심 데이터나 정책을 표현하는가? | 도메인 레이어 후보. |
| 비즈니스와 무관하게 재사용되는가? | 공통 기반 후보. |
| 상위 레이어를 알아야만 동작하는가? | 레이어 분리 실패 가능성. |

## FSD와의 관계
FSD는 레이어드 아키텍처를 프론트엔드에 맞게 구체화한 패턴이다.

| 일반 레이어 사고 | FSD 예시 |
|---|---|
| 앱 조립 | `app` |
| 화면 | `pages` |
| 큰 UI 블록 | `widgets` |
| 사용자 기능 | `features` |
| 도메인 데이터 | `entities` |
| 공통 기반 | `shared` |

하지만 모든 프로젝트가 FSD 레이어명을 그대로 써야 하는 것은 아니다. Obsidian에서는 FSD를 패턴으로 설명하고, 채택 여부는 프로젝트 `docs/`에서 정한다.

## 에이전트 적용 기준
- 먼저 프로젝트 `docs/`에서 실제 레이어 규칙을 확인한다.
- 프로젝트 규칙이 없으면 상위가 하위만 의존한다는 원칙으로 구조를 제안한다.
- 하위 레이어가 라우터, 페이지, 화면 컴포넌트를 import하면 경계 위반 후보로 본다.
- 레이어 이름보다 변경 이유와 재사용 범위를 우선한다.
- FSD 문서를 참고할 수 있지만, 프로젝트 채택 문서 없이 FSD를 강제하지 않는다.

## 안티패턴
- 하위 공통 모듈이 상위 화면 상태를 import한다.
- 모든 코드를 `components`, `hooks`, `utils`로만 나눈다.
- 레이어는 있지만 아무 방향 규칙이 없다.
- 도메인 로직이 UI 이벤트 핸들러 안에만 존재한다.
- 프로젝트마다 다른 레이어 규칙을 Obsidian 문서 하나로 강제한다.
