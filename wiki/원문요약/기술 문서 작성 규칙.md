## 1. 문서 유형 선택 (overall)

> 원문: `raw/toss/기술 문서 작성 규칙/overall.md`

문서 작성 전 목표·독자·상황에 맞는 유형을 먼저 선택한다.

| 유형 | 목적 |
|---|---|
| **학습 중심** | 개념/기술을 처음 배우는 독자 대상. 단계별 안내 + 실행 가능한 예제 |
| **깊은 이해** | 원리·철학 이해. 배경 + 동작 방식 + 비교 + 시각 자료 |
| **문제 해결** | 특정 문제/작업 해결. 원인 → 단계별 해결 → 코드 예제 |
| **참조** | 빠른 탐색용. 간결한 개요 + 구문/파라미터 + 예제 |

### 학습 중심 문서 필수 요소
1. 명확한 학습 목표 및 완료 후 얻게 될 능력
2. 사전 준비 사항 및 환경 설정
3. 단계별 안내 (무엇을, 왜 하는지)
4. 실행 가능한 코드 예제 (쉬운 것 → 점진적 난이도)
5. FAQ / 자주 발생하는 문제와 해결책

### 깊은 이해 문서 필수 요소
1. 기술/개념 등장 배경과 해결하려는 문제
2. 기본 원리와 동작 방식 상세 설명
3. 다른 접근 방식과 비교 및 장단점
4. 시각 요소 (다이어그램, 흐름도)
5. 실제 사용 사례 및 응용

### 문제 해결 문서 필수 요소
1. 명확한 문제 상황 또는 작업 목표
2. 문제 원인 또는 배경지식
3. 단계별 해결 방법
4. 실행 가능한 코드 예제/명령어
5. 환경별 차이점 (OS, 라이브러리 버전)
6. 해결책이 어떤 원리로 문제를 해결하는지

### 참조 문서 필수 요소
1. 간결한 개요 및 주요 기능
2. 구문 및 파라미터 (타입, 기본값, 필수 여부)
3. 반환값 및 타입
4. 사용 예제 (기본 → 다양한 활용)
5. 관련 API/함수/컴포넌트 연계
6. 주의 사항 및 제한사항

## 2. 문서 구조 원칙 (architecture)

> 원문: `raw/toss/기술 문서 작성 규칙/architecture.md`

1. **한 페이지 = 하나의 목표** — H4(####) 이상 깊이면 분리 신호
2. **개요 필수** — 독자가 "이 문서를 읽으면 무엇을 얻는가"를 첫 번째로 파악
3. **예측 가능한 구조** — 동일 수준 제목은 일관된 패턴, 기본 개념 → 상세 순서
4. **가치를 먼저** — 기능/세부 설정보다 독자가 얻는 이점을 도입부에
5. **효과적인 제목** — 핵심 키워드 포함, 30자 이내, 일관된 스타일(동사형 or 명사형), 평서문

## 3. 문장 작성 원칙 (editing)

> 원문: `raw/toss/기술 문서 작성 규칙/editing.md`

1. **필요한 정보만** — 한 문장 = 하나의 생각. 불필요하게 길거나 복잡한 문장 제거
2. **메타 담화 최소화** — "앞서 설명했듯이", "여러분도 아실 것입니다" 같은 표현 제거
3. **구체적으로** — 모호한 표현 대신 직접적 언어. 동사로 명확한 행동 지시
4. **일관되게** — 동일 개념을 다양한 표현으로 쓰지 않음. 약어는 첫 등장 시 풀어서 표기
5. **주체를 분명히** — 수동형 → 능동형. 도구가 주체가 아니라 독자(개발자)가 주체
