## 핵심 요약
**좋은 프론트엔드 코드 = 변경하기 쉬운 코드.** 새 요구사항 구현 시 기존 코드 수정·배포가 수월한 코드. 변경 용이성을 **4가지 기준**으로 판단:

| 기준 | 정의 |
|---|---|
| [[가독성]] (가독성) | 코드가 읽기 쉬운 정도. 한 번에 고려할 맥락이 적고 위→아래로 읽힘 |
| [[예측 가능성]] (예측 가능성) | 이름·파라미터·반환값만 보고 동작을 예측 가능 |
| [[응집도]] (응집도) | 함께 수정돼야 할 코드가 항상 같이 수정됨 |
| [[결합도]] (결합도) | 코드 수정 시 영향 범위가 작음 |

## 주요 주장 (트레이드오프)
- **4기준을 동시에 다 충족하기 어렵다.** 상황에 따라 우선순위를 선택해야 함.
- 가독성 ↔ 응집도: 공통화·추상화하면 응집도↑이나 가독성↓. 위험 높으면 응집도 우선, 낮으면 가독성 우선(중복 허용).
- 응집도 ↔ 결합도: 중복 허용하면 결합도↓이나, 한쪽 수정 시 다른 쪽 누락으로 응집도↓.

## 실무에서 가져갈 점
- 코드 리뷰에서 "왜 이 코드가 별로인지"를 4기준으로 객관화·언어화.
- 팀 공통 코드 품질 기준 수립의 공용어.
