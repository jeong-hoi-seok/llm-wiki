# AGENTS.md

이 파일은 모든 AI 에이전트가 따라야 하는 표준 운영 규칙이다.
Claude, Codex, Cursor, ChatGPT, 기타 에이전트는 이 파일을 우선 읽고 작업한다.

`CLAUDE.md`는 Claude 전용 진입 파일일 뿐이며, 상세 규칙은 이 `AGENTS.md`를 기준으로 한다.

## 목적

이 저장소는 프론트엔드(웹·앱) 개발 지식을 Markdown 기반 LLM Wiki로 관리하는 공간이다.

- 사람은 `raw/`에 원본 자료를 추가한다.
- AI 에이전트는 `raw/`를 읽고 `wiki/`에 정리된 지식 문서를 만든다.
- `context/`는 사용자 말투·코드 스타일·프로필처럼 잘 바뀌지 않는 개인 기준을 보관한다.
- Obsidian은 특정 프로젝트 규칙 저장소가 아니라, 프로젝트 `docs/` 문서 품질을 높이는 공통 프론트엔드 지식 베이스다.

> 방식 출처: Andrej Karpathy, LLM Wiki gist
> https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

## 기본 원칙

- `raw/` 안의 원본 자료는 수정하지 않는다. source of truth다.
- 에이전트는 기본적으로 `wiki/` 안의 문서만 생성하거나 수정한다. 사용자가 개인 기준 정리를 요청한 경우에만 `context/`를 생성하거나 수정한다.
- 답변과 문서는 기본 한국어로 작성한다.
- 중요한 개념은 `[[Wikilink]]`로 연결한다.
- 문서에는 가능한 한 출처를 남긴다.
- 사실과 추론을 구분한다.
- 확실하지 않은 내용은 `[검증 필요]`로 표시한다.
- 추론한 내용은 `[추론]`으로 표시한다.
- 새 자료를 처리하면 `wiki/index.md`와 해당 날짜의 `wiki/log/YYYY-MM-DD.md`를 갱신한다.
- 프로젝트별 실제 규칙은 각 프로젝트의 `docs/`, `AGENTS.md`, `CLAUDE.md`가 우선한다. 이 위키는 공통 지식과 판단 기준이다.

## 폴더 역할

### `raw/` — 원본 보관소

원본 자료를 보존하는 곳이다. LLM은 읽기만 한다.

- 블로그
- 공식 문서
- README
- 디자인 시스템 문서
- 기술 인터뷰
- 회의록 원본
- 외부 자료 복사본

현재 주요 원본:

- `raw/expo/pages/` — Expo 공식 문서 `docs/pages`의 md/mdx 원문
- `raw/feature-sliced-design/kr/docs/` — Feature-Sliced Design 공식 문서 한국어판 원문
- `raw/kakaopay/` — 카카오페이 공통 컴포넌트 원문
- `raw/testing-sources/` — 프론트엔드 테스트 자료 URL 목록
- `raw/toss/` — 토스 frontend-fundamentals 원문

### `wiki/` — LLM 정리본

LLM이 읽기 좋게 재구성한 지식층이다.

- `원문요약/` — 다중 파일·장문 raw의 압축 진입점, 출처 귀속 앵커. **조건 충족 시에만 생성**(Ingest 흐름 참고)
- `기술개념/` — 프론트엔드 기술 개념 사전. 하위는 `web/`(브라우저·DOM·Next.js·Tailwind·웹 번들러)과 `app/`(React Native·Expo·NativeWind·네이티브 런타임)으로 나눈다.
- `아키텍처/` — 아키텍처·폴더 구조 방법론
- `개발원칙/` — 코드 품질·접근성·디버깅 같은 개발 원칙
- `최적화/` — 성능·리소스·렌더링 비용 최적화 기준
- `테스트/` — 단위·통합·시각적 회귀 테스트와 Storybook 테스트 환경 기준
- `컨벤션/` — 네이밍·커밋·MR/PR 같은 작성 규칙
- `디자인시스템/` — 디자인 시스템·컴포넌트 설계
- `기술결정/` — 프로젝트와 무관하게 재사용 가능한 기술 판단 기준
- `기술비교/` — 기술·패턴 비교 문서. 필요 시 생성
- `첨부자료/` — 이미지·다이어그램 등 위키 첨부 자료. 필요 시 생성
- `템플릿/` — 문서 템플릿
- `index.md` — 전체 목차. 질문 답변 전 먼저 읽는다
- `log.md` — 작업 이력 색인
- `log/` — 일자별 작업 이력. `YYYY-MM-DD.md` 형식, append-only

### `context/` — 개인 정적 기준

사용자 개인의 잘 바뀌지 않는 기준을 보관하는 곳이다. 프로젝트 지식이 아니라 에이전트가 응답, 글쓰기, 코드 작성 톤을 맞추기 위해 참고하는 기준이다.

- `나의 글쓰기 스타일.md` — 사용자 목소리를 유지하기 위한 글쓰기 기준
- `원칙과 사고방식.md` — 사실성, 검증, 메트릭, 피드백 태도 기준
- `코드 스타일.md` — 코드 작성 취향과 스타일. 필요 시 생성
- `사용자 프로필.md` — 사용자에 대한 장기 맥락. 필요 시 생성

글을 대신 쓰거나 다듬는 작업은 `context/나의 글쓰기 스타일.md`를 우선 확인한다.
사용자 기여, 의사결정, 성과, 회고, 포트폴리오성 글을 작성할 때는 `context/원칙과 사고방식.md`도 함께 확인한다.
`context/` 변경은 개인 기준 변경이므로 `wiki/index.md`에는 넣지 않고, 필요한 경우 `wiki/log/YYYY-MM-DD.md`에만 기록한다.

## 작업 흐름

### Context First

사용자의 말투, 글쓰기, 판단 기준이 결과물에 영향을 주는 작업은 `context/`를 먼저 읽고 진행한다.

특히 아래 작업은 `wiki/index.md`나 관련 wiki 문서보다 `context/`를 우선 확인한다.

- 사용자를 대신해 글을 쓰거나 다듬는 작업
- wiki 문서 생성·수정·요약처럼 문체와 정보 밀도가 중요한 작업
- 피드백, 회고, 기여 정리, 의사결정 설명처럼 사용자의 사고방식이 드러나야 하는 작업
- 장문의 답변, 제안서, 문서화, 발표·포트폴리오성 글 작성

우선 확인 기준:

1. 기본 글쓰기·문체는 `context/나의 글쓰기 스타일.md`를 따른다.
2. 사실성, 검증 태도, 피드백 방식은 `context/원칙과 사고방식.md`를 따른다.
3. 코드 작성 취향이 필요한 경우 `context/코드 스타일.md`가 있으면 확인한다.
4. 사용자 프로필이 필요한 경우 `context/사용자 프로필.md`가 있으면 확인한다.

`context/`와 일반 템플릿이 충돌하면 `context/`를 우선한다. 단, 출처 표기, 사실·추론 구분, raw 보존 원칙은 유지한다.

### Ingest

`raw/`에 새 자료가 추가되면:

1. 원본을 읽는다.
2. 관련 `wiki/기술개념/`, `아키텍처/`, `개발원칙/`, `최적화/`, `테스트/`, `디자인시스템/` 문서를 새로 만들거나 갱신한다. **이것이 ingest의 본체다.**
3. 개념 문서 frontmatter `source:`에 `raw/` 경로, URL, 기준일을 반드시 남긴다. (출처 앵커는 개념 문서가 1차로 진다.)
4. 필요 시 `기술비교/`, `기술결정/` 문서를 만든다.
5. 관련 개념을 `[[Wikilink]]`로 연결한다.
6. `wiki/index.md`를 갱신한다.
7. 해당 날짜의 `wiki/log/YYYY-MM-DD.md`에 처리 내역을 추가한다.

`wiki/원문요약/`은 **항상 만들지 않는다. 아래 조건 중 하나라도 맞을 때만 만든다.**

| 원문요약 만든다 | 안 만든다 |
|---|---|
| raw가 다중 파일·장문이라 진입점 압축이 필요 (예: 토스 FF 134파일, 성능 체크리스트) | raw가 단일·간결한 md라 그 자체로 읽힘 |
| "어느 원문이 한 주장인지" 역추적·출처 귀속이 중요 | 개념 문서 frontmatter `source:`로 출처 추적이 충분 |
| 원문 주장과 우리 종합 지식을 분리 보존할 가치가 있음 | 원문 ↔ 개념이 1:1이라 분리 의미 없음 |

원문요약을 만들 때 규칙:

- **개념 문서 내용을 재진술하지 않는다.** 표·세부 설명은 개념 문서에만 둔다.
- 원문요약은 **출처 메타(URL·기준일·raw 경로) + 1문단 TL;DR + 개념 문서로의 `[[Wikilink]]` 포인터**로 슬림하게 유지한다.
- 같은 사실을 원문요약과 개념 문서 두 곳에서 유지보수하게 만들지 않는다.

### Query

사용자가 질문하면:

1. `wiki/index.md`를 먼저 읽는다.
2. 관련 wiki 문서를 찾는다.
3. 필요 시 `raw/` 원문을 확인한다.
4. 근거 문서를 명시하고, 사실과 추론을 구분한다.
5. 답변 자체를 별도 `wiki/질문답변/` 파일로 저장하지 않는다.
6. 해당 날짜의 `wiki/log/YYYY-MM-DD.md`에 query 기록을 추가한다.

### Commit

사용자가 커밋을 요청하면:

1. 먼저 staged 변경 범위를 확인한다.
2. 저장소의 커밋 컨벤션 문서를 찾고 읽는다.
   - 이 저장소에서는 `wiki/컨벤션/커밋 컨벤션.md`를 최우선으로 확인한다.
3. 커밋 컨벤션 문서가 있으면 그 문서를 최우선 기준으로 커밋 메시지를 작성한다.
4. 커밋 컨벤션 문서를 찾지 못한 경우에만 일반 Conventional Commit 규칙을 사용한다.
5. 커밋 전에 staged diff와 커밋 메시지가 컨벤션에 맞는지 확인한다.

### Lint

정기 점검 항목:

- 끊어진 `[[링크]]`
- 고립 문서
- 중복 문서
- 오래된 주장
- 충돌하는 주장
- 출처 없는 주장
- `index.md` 누락된 핵심 문서
- 너무 길거나 짧은 문서

수정 가능한 것은 수정한다.
판단 필요한 것은 해당 날짜의 `wiki/log/YYYY-MM-DD.md`에 기록한다.

## 문서 작성 규칙

템플릿은 `wiki/템플릿/` 아래 문서를 우선 사용한다.

기본 구조:

```md
# 제목

## 한 줄 요약

## 핵심 내용

## 왜 중요한가

## 실무 적용 기준

## 예시

## 관련 문서
- [[]]

## 출처
-
```

## 제목과 파일명 규칙

- 파일명은 한국어 사용자가 바로 이해할 수 있게 쓴다.
- 완전 번역보다 직관성이 우선이다.
- 기술 고유명은 유지한다.
- 예: `FSD 폴더구조`, `코드 품질`, `공통 컴포넌트 추출 기준`, `Tailwind CSS`, `Next.js`, `shadcn-ui`
- 새 파일을 만들면 H1 제목도 파일명과 맞춘다.

## 링크 규칙

- 중요한 개념은 `[[링크]]`로 연결한다.
- 너무 일반적인 단어는 링크하지 않는다.
- 존재하지 않는 예시 링크를 만들지 않는다. 예시는 코드 블록이나 백틱 텍스트로 쓴다.
- 좋은 예: `[[FSD 폴더구조]]`, `[[헤드리스 컴포넌트]]`, `[[Tailwind CSS]]`
- 나쁜 예: `중요`, `나중에`, `참고`

## Frontmatter

기본 형식:

```md
---
type: concept | source-summary | decision | comparison | architecture | overview | index | context
scope: <문서가 다루는 구체 범위>
platform: common | web | app | [web, app]
status: draft | summarized | verified | deprecated
tags: [frontend]
source: <raw/ 경로 또는 URL>
created: 2026-06-19
updated: 2026-06-19
---
```

`platform` 값 기준:

- `common` — 웹·앱 공통 적용
- `web` — Next.js 등 웹 전용
- `app` — React Native·Expo 전용
- `[web, app]` — 양쪽 사용하나 구현 방식이 다른 문서

`scope` 값 기준:

- 문서가 적용되는 범위를 짧고 구체적인 kebab-case로 쓴다.
- `pattern`, `guide`, `frontend`처럼 넓은 값만 단독으로 쓰지 않는다.
- 좋은 예: `fsd-architecture-pattern`, `feature-sliced-design-official-docs-kr`, `writing-style`
- `context/` 문서도 개인 기준의 적용 범위를 알 수 있게 `scope`를 둔다.

`source`에 여러 wiki 링크를 넣을 때는 Obsidian Properties가 깨지지 않게 YAML 리스트 문자열로 쓴다.

```yaml
source:
  - "[[코드 품질]]"
  - "[[응집도]]"
```

## 플랫폼 참조 규칙

- 프로젝트에 `platform:` 선언이 있으면 해당 플랫폼과 `common` 문서를 우선 참조한다.
- 웹 프로젝트는 `web` + `common`을 우선 참조한다.
- 앱 프로젝트는 `app` + `common`을 우선 참조한다.
- 플랫폼이 불명확하면 `common` 문서부터 읽고 필요한 경우 웹·앱 문서를 추가 확인한다.

## Obsidian과 프로젝트 docs 역할 분리

Obsidian wiki:

- 순수 프론트엔드 아카이브
- 개념, 원칙, 패턴, 선택 기준
- 어떤 프로젝트에도 직접 강제하지 않음
- 프로젝트 `docs/` 품질을 높이는 참고 지식

프로젝트 `docs/`:

- 특정 프로젝트의 실제 규칙
- 폴더 구조, 스펙, 컨벤션, 의사결정
- 에이전트가 실제로 따라야 하는 실행 기준

충돌 시 프로젝트 `docs/`가 우선한다.

## 금지 사항

- `raw/` 원본 수정 금지
- 출처 없는 주장을 사실처럼 쓰지 않기
- 확실하지 않은 내용을 단정하지 않기
- 같은 개념 문서를 중복 생성하지 않기
- 원문요약에 개념 문서 내용을 재진술해 같은 사실을 두 곳에서 유지보수하게 만들지 않기
- 존재하지 않는 위키링크를 남기지 않기
- `index.md` 갱신 누락 금지
- 해당 날짜의 `log/YYYY-MM-DD.md` 갱신 누락 금지

## 작업 후 검증

작업 후 가능한 범위에서 확인한다.

- `obsidian read path="frontend-llm-wiki/wiki/index.md"`가 성공하는지
- 새로 만든 문서가 `obsidian read`로 읽히는지
- `[[링크]]`가 실제 파일명과 맞는지
- `index.md`에 핵심 문서가 누락되지 않았는지
- `wiki/log/YYYY-MM-DD.md`에 작업 기록이 남았는지
